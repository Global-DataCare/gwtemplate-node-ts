# Organization Registration Process Documentation

This document provides a comprehensive overview of the organization registration process, including the data flow, component responsibilities, and usage as a prompt for developers.

## 1. Overview

The organization registration process allows administrators to register new organizations (tenants) within the system. The process involves collecting organization details, user information, and consent, then structuring this data and processing it on the backend. This document outlines the steps involved and the responsibilities of each component.

## 2. Data Flow

2.1. **Frontend Form:** The user interacts with a template-driven form on the frontend application to input organization details, administrator information, and consent.

2.2. **FAPI Envelope with DIDComm:** The external application encapsulates the collected data within a FAPI-compliant envelope, secured using DIDComm (Decentralized Identifiers and Decentralized Communication).
    *   **FAPI:** Financial-grade API (FAPI) is a set of security standards designed to protect sensitive financial data. Using FAPI ensures a high level of security for the registration process.
    *   **DIDComm:** DIDComm provides a secure, private, and verifiable communication channel between the frontend and backend. The data is encrypted and signed using DIDs, ensuring authenticity and integrity. The **`thid` (thread ID)**, used for tracking the request-response flow, is included *within* the DIDComm message itself. The `thid` is **unique to both the requester and the server *for a given interaction*, generated by the requester in a decentralized manner using a UUID v4**.

2.3. **API Endpoint:** The backend receives the FAPI/DIDComm envelope at a dedicated API endpoint (e.g., `.../host/cds-<jurisction>/v1/<sector>/registry/org.schema/Organization/_batch`).

2.4. **Envelope Decoding:** The backend decodes the FAPI envelope and decrypts the DIDComm message. The decrypted payload contains the registration data. This payload, which now includes the `thid` within, is considered the `job`.

2.5. **OrganizationRegistrationManager:** The API endpoint calls the `OrganizationRegistrationManager` to handle the registration process.

2.6. **Validation:** The `OrganizationRegistrationManager` validates the incoming data, ensuring that it conforms to the expected format.

2.7. **Claim Separation:** The `OrganizationRegistrationManager` separates the claims into a structured format, extracting different types of resources (e.g., Organization, Person, Service) based on claim prefixes.

2.8. **UUID Generation/Validation:** The `OrganizationRegistrationManager` generates unique identifiers (UUID v4) for the different resources.

2.9. **Data Persistence:** *In memory for demo purposes only.*

2.10. **Response:** The backend sends a response to the frontend, indicating the success or failure of the registration, and includes a `code` (authorization code) for retrieving an access token. The code flow is FAPI compliant, leveraging the DIDComm registry request `thid` (found in the decoded `job`) as source challenge similar to the OpenID PKCE flow, and preventing replay attacks.

## 3. Components

### 3.1 Frontend

*   **Responsibility:**
    *   Render form templates based on configuration.
    *   Collect user input.
    *   Construct the FAPI envelope.
    *   Encrypt and sign the data using DIDComm.
    *   Submit the FAPI/DIDComm envelope to the backend.
*   **Technology:** Any modern web framework (e.g., React, Angular, Vue.js) with DIDComm and FAPI libraries.
*   **Key Considerations:**
    *   Implement proper DIDComm key management and secure storage.
    *   Ensure FAPI compliance by following the relevant security standards.
    *   Focus on template rendering and data collection, minimizing complex logic.

### 3.2 API Endpoint

*   **Responsibility:**
    *   Receive the FAPI/DIDComm envelope.
    *   Validate the FAPI envelope.
    *   Decrypt and verify the DIDComm message.
    *   Call the `OrganizationRegistrationManager`.
    *   Handle request and response lifecycle.
    *   Handle authentication and authorization (FAPI compliant).
*   **Technology:** Node.js with Express, Koa, or similar, along with DIDComm and FAPI libraries.
*   **Key Considerations:**
    *   Implement proper FAPI validation.
    *   Handle DIDComm key resolution and verification.
    *   Implement robust authentication and authorization.
    *   Handle errors gracefully and provide informative responses.

### 3.3 OrganizationRegistrationManager

*   **Responsibility:**
    *   Validate the incoming data (the decrypted DIDComm payload).
    *   Separate claims into different resource types.
    *   Generate unique identifiers (UUID v4) for each resource.
    *   Orchestrate the data persistence (in future implementations).
*   **Technology:** TypeScript/JavaScript.
*   **Key Methods:**
    *   `registerOrganizations(job: any, environment?: string): Promise<any>`: Orchestrates the whole process. The `job` parameter represents the decrypted DIDComm payload.
*   **Key Considerations:**
    *   Implement robust validation to ensure data integrity.
    *   Handle the logic for UUID v4 generation/validation.
    *   Provide clear and informative error messages.

## 4. Data Structures

### 4.1 Input (FAPI/DIDComm Envelope)

The API endpoint receives a FAPI-compliant envelope that contains a DIDComm message. The DIDComm message contains the registration data, encrypted and signed for security.

### 4.2 Job (Decoded DIDComm Payload)

After decoding the FAPI envelope and decrypting the DIDComm message, the backend obtains the `job` object. The `job` object contains the registration data. Each "entry" of the batch request to register organizations will have this structure:

The `job` object follows this structure (the external app generates and sends it enveloped in a DIDComm message):

```json
{
  "thid": "<uuid>",
  // ... other data in the payload (meta.jws.protected, meta.jwe.header, aud, nbf, exp, etc.) ...
  "body": {
    // Interoperable JSON:API Primary Document (or FHIR Bundle when supported)
    "data": [
      {
        "type": "template", // Fixed to "template" in this implementation
        "id": "string",     // Internal ID for the top-level data in this entry (the organization)
        "meta": {
          "templateId": "OrgRegistrationForm",
          "templateVersion": "1.0",
          "claims": {
            "@claims": "org.schema",
            "@type": "template",
            "org.schema.Organization.legalName": "Example Corp",
            "org.schema.Organization.addressCountry": "US",

## 5. Code Examples (Simplified)

### 5.1 TypeScript (OrganizationRegistrationManager)

```typescript
import { v4 as uuidv4, validate as uuidValidate } from 'uuid';

async registerOrganizations(job: any, environment?: string): Promise<any> {
  // ... (validation and data extraction) ...

  const includedResources = [];
  for (const type of resourceTypes) {
    let resourceId: string;
    const identifierClaim = `org.schema.${type}.identifier`;
    if (claims[identifierClaim]) {
      const identifier = claims[identifierClaim];
      if (environment !== "demo" && uuidValidate(identifier)) {
        resourceId = identifier;
      } else {
        resourceId = uuidv4();
      }
    } else {
      resourceId = uuidv4();
    }

    includedResources.push({
      type: type,
      id: resourceId,
      meta: {
        claims: resourceClaim
      }
    });
  }

  // Construct the JSON:API Primary Document
  const templateId = uuidv4();
  const result = {
    data: [{
      type: "template",
      id: templateId,
      meta: {
        templateId: job.body.data[0].meta.templateId, // Or however you access it
        templateVersion: job.body.data[0].meta.templateVersion,
        claims: claims
      },
      relationships: {
        organization: { data: { type: "Organization", id: includedResources[0]?.id || null }},
        person: { data: { type: "Person", id: includedResources[1]?.id || null }},
        service: { data: { type: "Service", id: includedResources[2]?.id || null }},
      }
    }],
    included: includedResources
  };

  return result;
}
```

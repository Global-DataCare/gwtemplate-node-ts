# Deployment Guide: Google Cloud Run

This guide provides instructions for deploying the application to Google Cloud Run using the provided automation script.

## Overview

This project includes an automated script (`cloud_deploy.sh` in the root directory) to build and deploy the application as a containerized service on Google Cloud Run. This is ideal for creating staging, testing, or production environments.

The process is designed to be simple and repeatable, relying on a local `.env` file for configuration.

---

## 1. Initial Setup (One-Time Configuration)

Before you can deploy, you need to ensure your local environment is correctly authenticated and your Google Cloud project is properly configured.

### a. Authenticate Your Local `gcloud` SDK

Your local machine needs the correct credentials to interact with your GCP project.

1.  **Log in to Google Cloud:**
    ```bash
    gcloud auth login
    ```
    This will open a browser window for you to log in with the Google account that has access to your project.

2.  **Set up Application Default Credentials:**
    ```bash
    gcloud auth application-default login
    ```

3.  **Set Your Project:**
    Replace `[YOUR_GCP_PROJECT_ID]` with your actual project ID (e.g., `globaldatacare-test`).
    ```bash
    gcloud config set project [YOUR_GCP_PROJECT_ID]
    ```

4.  **Set the Quota Project:**
    This syncs your application credentials with your active project to avoid warnings.
    ```bash
    gcloud auth application-default set-quota-project [YOUR_GCP_PROJECT_ID]
    ```

### b. Grant Permissions to the Cloud Run Service Account

By default, a new Cloud Run service does not have permission to access other Google Cloud services like Firestore or Cloud Storage. You must grant these permissions manually.

1.  **Identify the Service Account:** Cloud Run uses the **Compute Engine default service account** by default. Its email address looks like this: `[PROJECT_NUMBER]-compute@developer.gserviceaccount.com`.

2.  **Navigate to IAM:** In the Google Cloud Console, go to the "IAM & Admin" > "IAM" page.

3.  **Grant Roles:**
    *   Find the "Compute Engine default service account" in the list of principals.
    *   Click the pencil icon (Edit) to modify its roles.
    *   Click **"+ ADD ANOTHER ROLE"** and add the following two roles:
        *   `Cloud Datastore User`: Provides access to Firestore.
        *   `Storage Object Admin`: Provides access to Google Cloud Storage.
    *   Click **SAVE**.

This step is crucial. Without these permissions, the container will fail to start on Cloud Run.

## 2. The Deployment Process

Once the initial setup is complete, deploying is a simple, repeatable process.

### a. Configure Your Deployment Environment

1.  **Copy the template:** `cp env.example .env`
2.  **Edit `.env`:** Fill in the variables under the `GCP Deployment Configuration` section. For a full explanation of all variables, see the **[Setup Guide](01.D-SETUP-GUIDE.md)**.

### b. Run the Script

1.  **Make the script executable** (only needs to be done once):
    ```bash
    chmod +x cloud_deploy.sh
    ```

2.  **Execute the deployment:**
    ```bash
    ./cloud_deploy.sh
    ```

The script will now run all the pre-flight checks and deployment steps automatically. Upon completion, it will output the public URL of the service.

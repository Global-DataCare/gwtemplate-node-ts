# Guide: Implementing a New Asynchronous Endpoint

This document is the canonical "cookbook" for adding a new, secure asynchronous endpoint. Following these patterns is essential for system consistency.

## Architectural Pattern: Asynchronous Job Submission

The system uses a controller-led, non-blocking pattern for sensitive operations.

1.  **`parseCdsRequest` Middleware:** Parses the URL into a `req.cdsRequest` object, which also contains the original request headers.
2.  **Controller:** Orchestrates the submission by validating the request, calling the `kmsService` to handle security, creating a job, and adding it to the queue.
3.  **Worker & Manager:** A background worker routes the job to the appropriate Manager, where the business logic is executed and the response is formatted according to content negotiation rules.

## Step-by-Step Implementation Guide

### Step 1: Implement Business Logic (TDD in a Manager)

All business logic **MUST** be implemented and unit-tested in a Manager class first. The manager's `process(job)` method is the entry point and will contain a `switch` statement based on `job.action` to route to the correct internal logic.

Crucially, the manager is also responsible for **Content Negotiation** for the final response (see detailed section below).

### Step 2: Create the Controller

The Controller is the HTTP entry point.

**File Location:** `src/routes/handlers/{section}/ExampleController.ts`

```typescript
// Example: src/routes/handlers/example/ExampleController.ts
import { NextFunction, Request, Response } from 'express';
// ... other necessary imports

export class ExampleController {
  private queueAdapter: QueueAdapter;
  private kmsService: IKmsService;

  constructor(queueAdapter: QueueAdapter, kmsService: IKmsService) {
    this.queueAdapter = queueAdapter;
    this.kmsService = kmsService;
  }

  public async myActionHandler(req: Request, res: Response, next: NextFunction): Promise<void> {
    try {
      // req.cdsRequest is populated by the parseCdsRequest middleware
      const { sector, resourceType, tenantId, action } = req.cdsRequest!;
      
      // 1. Validation logic here...
      if (resourceType !== 'ExampleResource') {
        throw new ManagerError(/* ... */);
      }
      
      // 2. Decode & Verify Security
      const decodedJob = await this.kmsService.decodeJobRequest(req.body.request);
      
      // 3. Prepare & Enqueue Job
      const vaultId = getTenantVaultId(sector!, tenantId!);
      const jobName = createJobName(vaultId, resourceType!, action!);
      
      // IMPORTANT: Pass the original headers to the job for content negotiation
      decodedJob.content.headers = req.headers; // TODO: REVIEW as decodedJob.meta.headers
      const job: JobRequest = { ...req.cdsRequest!, ...decodedJob };

      await this.queueAdapter.addJob(jobName, job);
      
      // 4. Respond 202 Accepted
      const pollingUrl = req.originalUrl.replace(`/${action}`, '/_batch-response');
      res.location(pollingUrl).status(202).send(); // Body MUST be empty
    } catch (error) {
      next(error);
    }
  }
}
```

### Step 3: Create the Router

The Router maps the URL to your controller.

**File Location:** `src/routes/example.ts`

```typescript
// Example: src/routes/example.ts
import { Router } from 'express';
// ... other necessary imports

export function createExampleRouter(
  queueAdapter: QueueAdapter,
  kmsService: IKmsService
): Router {
  const router = Router();
  const controller = new ExampleController(queueAdapter, kmsService);

  router.post(
    '/:tenantId/cds-:jurisdiction/v1/:sector/example/org.schema/ExampleResource/_myAction',
    parseCdsRequest,
    controller.myActionHandler.bind(controller)
  );

  return router;
}
```

### Step 4: Wire the Router in `server.ts`

Import and mount your new router in `server.ts`.

```typescript
// In src/server.ts
import { createExampleRouter } from './routes/example';

// ... inside startServer() ...
  const exampleRouter = createExampleRouter(queueAdapter, kmsService);
  app.use('/', exampleRouter);
```

## The Asynchronous Response Pattern

### The `202 Accepted` Response

The initial response **MUST** adhere to the following rules:

-   **Status Code:** `202 Accepted`
-   **Body:** The response body **MUST be empty**.
-   **`Location` Header:** Contains the polling URL, constructed by replacing the action segment (e.g., `/_discovery`) with `/_batch-response`.

### The Final `200 OK` Response (Content Negotiation & Polling Handler Logic)

The format of the final, successful response is determined by **Content Negotiation** at the time the job is processed, and the `pollingHandler` is responsible for delivering this final format correctly.

> **Important Note on Plaintext vs. Encrypted Flows:**
> The system follows a critical security rule: **all job results are stored encrypted in the response store**.
> - For a standard **secure flow** (where the client sends an encrypted JWE), the `pollingHandler` returns the resulting JWE directly.
> - For a **plaintext flow** (where the client sends `Content-Type: application/json`), the `Worker` encrypts the result using the tenant's own public key (an "encrypt-to-self" pattern). The `pollingHandler` **is responsible for decrypting this result** before sending the final plaintext JSON back to the client. This ensures clients that send JSON receive JSON.

#### 1. Tenant Configuration

A tenant's supported plaintext "legacy" output formats are configured via the claim:

-   **Claim:** `org.schema.Service.serviceOutput.additionalType`
-   **Value:** An array of supported MIME types (e.g., `['application/fhir+json', 'application/json']`).

The default `'DIDComm'` (secure JWE) flow is **always** supported as a fallback.

#### 2. Manager Logic

The business logic `Manager` **MUST** perform the following steps to format the response:

1.  **Get Tenant Config:** Retrieve the `serviceOutput.additionalType` array for the tenant. A helper on the `TenantsCacheManager` should be used for this.
2.  **Get `Accept` Header:** Inspect the `Accept` header from the original request, which is passed in `job.input.headers`.
3.  **Determine Format:**
    *   If the client's `Accept` header requests a MIME type that is present in the tenant's allowed list, the manager generates a plaintext response in that format.
    *   Otherwise, the manager **MUST** default to the secure `'DIDComm'` (JWE) format.

#### 3. Response Formats

##### **Default Flow: DIDComm (Secure JWE)**

-   **`Content-Type`:** `application/x-www-form-urlencoded`
-   **Envelope:** A JWE-wrapped, signed `IDecodedDidcommPayload` (e.g., `response=ey...`).
-   **Payload:** The `Bundle` object uses the key `"data"` for its results array.

##### **Legacy Flow: FHIR Conformance**

-   **Trigger:** Tenant allows `application/fhir+json` AND client sends `Accept: application/fhir+json`.
-   **`Content-Type`:** `application/fhir+json`
-   **Envelope:** The **plaintext, unwrapped** FHIR Bundle.
-   **Payload:** A valid FHIR Bundle using `"entry"` for its results array.

##### **Legacy Flow: JSON:API**

-   **Trigger:** Tenant allows `application/json` AND client sends `Accept: application/json`.
-   **`Content-Type`:** `application/json`
-   **Envelope:** The **plaintext, unwrapped** JSON:API-style Bundle.
-   **Payload:** The `Bundle` object uses the key `"data"` for its results array.

## Core Implementation Patterns

### Authorization via `didConfig`

Before queueing a job, the system **MUST** verify that the requested action is permitted for the tenant. This is a critical **authorization** step that happens after authentication.

-   **Location:** This check should occur in a dedicated `authorizeRequest` middleware, after `parseCdsRequest` and `decodeRequest`.
-   **Logic:**
    1.  The middleware constructs the `vaultId` for the tenant.
    2.  It calls `tenantsCacheManager.getDidServiceConfig(vaultId)` to retrieve the array of configured services for that tenant.
    3.  It finds the service that matches the request's `sector`, `section`, and `format`.
    4.  It then checks if the requested `resourceType` and `action` are listed in that service's `serviceEndpoint` and `actions` arrays.
    5.  If the action is not explicitly allowed, the middleware **MUST** reject the request with a `403 Forbidden` or `404 Not Found` error.

This ensures that tenants have explicit control over which API features are enabled for them.

### Worker Processing Flow

The `Worker` is a simple router.

1.  It dequeues a `job` and `jobName`.
2.  It calls `parseJobName(jobName)` to extract the `resourceType`.
3.  A `switch (resourceType)` statement selects the correct Manager from the `managerRegistry`.
4.  It calls `manager.process(job)`, delegating all further logic.

### Error Handling within Managers

Error handling is **decentralized** and handled within each Manager to maintain business context.

-   The main processing loop (e.g., iterating a batch) **MUST** wrap each unit of work in a `try...catch` block.
-   The `catch` block **MUST** call a `private handleError(error, entry)` method within the same Manager.
-   This method inspects the error type:
    -   If it's a `ManagerError` (a predictable business error), it creates a structured error response (e.g., a `400` or `404`) using the `createOperationOutcome` utility.
    -   If it's an unexpected exception, it creates a generic `500 Internal Server Error` response.
-   This pattern ensures that one failed entry in a batch does not stop the processing of others.

## Future Architectural Improvement: Middleware Pipeline

To further streamline development and align with the pattern described above, the Controller logic should be broken down into a pipeline of distinct, reusable middlewares.

**Future Router:**
```typescript
router.post(
  '/.../_myAction',
  parseCdsRequest,           // 1. Parses URL
  decodeRequest(kmsService), // 2. Authenticates by decoding JWE/JWS
  authorizeRequest(tenantManager), // 3. Authorizes against tenant's didConfig
  submitJob(queueAdapter)      // 4. Submits the job
);
```

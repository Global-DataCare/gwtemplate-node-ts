# Tenancy and Vault Architecture

This document explains the architecture for multi-tenancy, data storage, and caching within the gateway. The central goal is to abstract the physical storage mechanism, allowing the application to run on different backends (in-memory, Firestore, etc.) without changing the business logic.

## Key Concepts & Vocabulary

A clear distinction between logical and physical identifiers is crucial:

-   **`vaultId` (Logical ID):** A composite, human-readable, unique string that identifies a tenant's logical data container (e.g., `'host'`, `'health-care_acme'`). This ID is used throughout the business logic layer (Managers) and serves as the **document ID** for a tenant's record within the Host's vault.
-   **`collectionName` (Physical ID):** The actual name of the collection in the underlying database (specifically Firestore). It's a long, deterministic, and non-human-readable string generated from a tenant's immutable claims (e.g., `ES_TAX_B12345678_health-care`). This is an **implementation detail** managed by the `TenantsCacheManager` and used by the `IVaultRepository`.

## The "Host-as-Registry" Pattern

The architecture is built on the principle that the **Host tenant acts as the master registry for all other tenants.**

-   The **Host's Vault** (e.g., the Firestore collection `ES_tax_B12345678_system`) contains a special section named `'tenants'`.
-   When a new tenant (e.g., `'acme'`) is created, its encrypted configuration document (`EntityConfig`) is stored inside the `'tenants'` section of the **Host's Vault**.
-   Each tenant also has its own separate vault for its own operational data (e.g., employees, customers), but its master configuration record lives with the Host.

This centralized registry is the key to discovering and managing all tenants in the system.

## The `TenantsCacheManager` Strategy

The `TenantsCacheManager` is the core component that bridges the logical and physical worlds. It implements a **bimodal caching strategy** depending on the configured `DB_PROVIDER`.

### Mode 1: `DB_PROVIDER=mem` (Eager Loading)

-   **When:** Used for local development and fast, isolated tests.
-   **How it Works:** On server startup, `loadTenants()` reads all tenant configurations from the `VaultMemRepository` (which are already in memory) and caches them.
-   **Why:** It's extremely fast and simple. All lookups are instant after startup, which is ideal for testing environments.

### Mode 2: `DB_PROVIDER=firestore` (Lazy Loading + Write-Through)

-   **When:** Used for E2E tests and production.
-   **How it Works:**
    1.  **Startup:** On server startup, `loadTenants()` is called, but in this mode, it only **preloads the Host's own configuration**. The server starts almost instantly.
    2.  **Cache Miss (Lazy Load):** The first time a service requests a tenant (e.g., `'acme'`), `getTenantByAlternateName('acme')` checks its in-memory cache. If not found, it performs a **single, targeted query** against the Host's vault (`'tenants'` section) to find the document for `'acme'`.
    3.  **Populate Cache:** Once found, the tenant's configuration is decrypted and stored in the in-memory cache.
    4.  **Cache Hit:** All subsequent requests for `'acme'` are served instantly from memory.
    5.  **Cache Write-Through:** When `HostingManager` creates a new tenant, after persisting it to the database, it immediately calls `tenantsCacheManager.setTenantConfig()` to add the new tenant to the cache. This prevents a "cache miss" right after creation.

-   **Why:** This strategy provides the best of both worlds for a production environment: extremely fast server startup times and excellent runtime performance (after the first hit) while minimizing database reads.

## Component Responsibilities

-   **`server.ts`:** Reads the `.env` file. Based on `DB_PROVIDER`, it injects either `FirestoreVaultRepository` or `VaultMemRepository` into the application. It then calls `tenantsCacheManager.loadTenants()` to kick off the caching process.
-   **API Routers:** Extract the `alternateName` from the URL path (e.g., `/acme/...`).
-   **`TenantsCacheManager`:** Acts as the central "translator" and single source of truth for tenancy information. It is the **only component responsible for translating a logical `vaultId` into a physical `collectionName`**. It manages the cache and implements the lazy/eager loading strategies.
-   **`IVaultRepository`:** The abstract storage interface. Its methods (`put`, `get`, `query`) are intentionally "dumb" and operate on physical `collectionName`s provided by the business layer. The methods `createNewVault` and `vaultExists` are exceptions that operate on logical `vaultId`s, as they pertain to the tenant's logical registration record within the Host's vault.
-   **`FirestoreVaultRepository`:** The concrete implementation. 
    -   `createNewVault` is a **no-op** (it does nothing) because the tenant's collection is created implicitly by the first `put` operation. The method exists to satisfy the interface contract defined by the `VaultMemRepository`.
    -   `vaultExists` translates its logical `vaultId` into a `get` operation against the Host's vault to check for the tenant's registration document.
-   **Business Managers** (e.g., `HostingManager`, `EmployeeManager`): Operate exclusively with the logical `vaultId`. Before performing any storage operation, they must ask the `TenantsCacheManager` for the correct `collectionName` to use for the `IVaultRepository` call.

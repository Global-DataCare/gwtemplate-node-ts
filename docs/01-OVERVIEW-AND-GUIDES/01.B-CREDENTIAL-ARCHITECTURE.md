# 01.B Verifiable Credential Architecture

This document outlines the core architectural principles and patterns for creating, identifying, and issuing Verifiable Credentials (VCs) within the system. These rules are mandatory to ensure interoperability, security, and deterministic behavior.

## 1. Verifiable Credential ID (`vc.id`)

The `id` field of a Verifiable Credential **MUST** be a deterministic, versioned identifier. It is generated based on the credential subject's public identifier and the issuance timestamp.

### Formula

The ID is generated using the following formula:

`z(multibase(multihash(SHA3-256(<URN>:timestamp:epoch:<value>))))`

Where:
-   **`<URN>`**: The stable URN of the credential subject, sourced from the `credentialSubject.identifier` field.
-   **`<value>`**: The Unix epoch timestamp (in seconds) derived from the `vc.validFrom` ISO 8601 date string.

This approach ensures that every version of a credential (as defined by its issuance date) has a unique, content-addressable ID.

## 2. Credential Subject Identifier

The public, stable identifier for the subject of a credential (e.g., an organization, person, or device) **MUST** be placed in the `credentialSubject.identifier` field.

-   **`credentialSubject.identifier`**: **Mandatory**. Holds the public URN or DID of the subject.
-   **`credentialSubject.id`**: **Prohibited**. This field should not be used for the primary identifier to avoid ambiguity with internal database IDs or context-specific identifiers. This aligns with standards like schema.org and FHIR where `.id` is often a local identifier.

## 3. Required Issuance Formats

The system, specifically the `CredentialManager` and higher-level managers, **MUST** be capable of issuing credentials in multiple formats to support various interoperability and security requirements.

### 3.1. Standard JSON-LD (`application/ld+json`)

-   **Description**: The default W3C format.
-   **Structure**: A standard JSON-LD object with a `proof` array.
-   **Signatures**: The `proof` array **MUST** support holding multiple, different signature types simultaneously (e.g., one Ecdsa signature and one ML-DSA signature).

### 3.2. Verifiable Credential JWT (`application/vc+jwt`)

-   **Description**: A JWT-encoded Verifiable Credential.
-   **Structure**: The VC claims are the payload of the JWT.
-   **Signatures**: The signature is part of the JWT structure itself. This format contains only a single signature.

### 3.3. JWS Detached (`application/jws`)

-   **Description**: A detached JSON Web Signature, where the payload is not included.
-   **Use Case**: Used in scenarios where the payload is large or transferred separately.

## 4. PDF417 Barcode Issuance Strategy

To support physical-to-digital verification, the system will implement a barcode issuance strategy with the following characteristics:

-   **Content**: The barcode (e.g., PDF417) will encode a JWS Detached or a minimal VC-JWT.
-   **Primary Signature**: The signature embedded in the barcode should be a widely supported, fast-to-verify algorithm like **Ecdsa**.
-   **Secondary Signature Resolution**: The payload of the barcoded credential will contain a URL or other resolver pointing to a more secure, post-quantum signature (e.g., **ML-DSA**). This allows a verifier to scan the barcode, quickly validate the primary signature, and then fetch the high-security signature from a trusted endpoint to complete a full, secure verification.

## Lifecycle 101

Status: Current guidance for GW CORE lifecycle semantics and SDK alignment.

This document is the local "101" for lifecycle semantics in GW CORE. It does not replace shared examples from `gdc-common-utils-ts`, but it explains the rules that SDK prompts and GW maintainers should treat as the stable model for this repository.

## Scope

This document covers:

- individual lifecycle in GW CORE
- employee lifecycle in GW CORE
- hosted tenant authorization lifecycle in GW CORE
- how `status`, `audit`, and blockchain-facing lifecycle projections should relate

This document does not define extension-domain actions for products, genomics, or other future repositories.

## Core rules

- `_search` uses search parameters or search claims.
- `_batch + POST` creates a resource.
- `_batch + PUT` updates resource data.
- `_batch + PATCH` performs lifecycle transitions or other partial operational updates.
- `/_purge + POST` is an explicit business command, not generic CRUD.
- `DELETE` must not be used as the semantic model for `disable`.

Mental model:

- change email, phone, name, occupation: `PUT`
- disable or enable a resource: `PATCH`
- mark a resource as archived or soft-deleted: `PATCH`
- release licenses and preserve traceability: `/_purge`

## Canonical storage model

- `ConfidentialStorageDoc.status` is the canonical current status.
- `ConfidentialStorageDoc.content` stores the business resource and claims.
- `ConfidentialStorageDoc.audit` stores metadata about the latest relevant change.
- blockchain should receive a derived lifecycle event, not the raw confidential storage document.

`status` is the source of truth. `audit` must describe the transition, not compete with `status`.

## Audit guidance

`audit.deactivated` is too weak for a lifecycle domain that can include:

- `pending`
- `active`
- `inactive`
- `purged`
- `suspended`
- `revoked`

GW CORE should keep `status` as the canonical field and extend `audit` with transition metadata instead of duplicating the current status value.

Recommended audit fields:

- `audit.updated`
- `audit.statusChangedAt`
- `audit.statusChangedBy`
- `audit.previousStatus`
- `audit.statusReason`

`audit.status` is not recommended as a primary field because it duplicates `status` and introduces drift risk.

## Derived blockchain event

When lifecycle changes must be anchored or projected to blockchain, the event should be derived from `status` plus audit transition fields.

Example:

```json
{
  "resourceId": "…",
  "resourceType": "…",
  "previousStatus": "active",
  "status": "inactive",
  "changedAt": "2026-05-26T12:00:00Z",
  "changedBy": "did:example:actor",
  "reason": "manual-disable"
}
```

Source mapping:

- `status` from `ConfidentialStorageDoc.status`
- `previousStatus` from `audit.previousStatus`
- `changedAt` from `audit.statusChangedAt`
- `changedBy` from `audit.statusChangedBy`
- `reason` from `audit.statusReason`

## Payload conventions

- Prefer `resource.id` as the resource locator when available.
- `resource.identifier` can be used when the domain contract defines it clearly.
- `resource.meta.claims` is the canonical carrier for resource business claims.
- Do not model `PATCH` or `/_purge` as internal `_search` operations with query-style URL parameters.

## GW CORE lifecycle semantics

### Individual / family

- `individual/org.schema/Organization/_disable`
  - disables the hosted individual/family registration
  - does not release licenses
- `individual/org.schema/Organization/_purge`
  - requires the registration to be inactive first
  - releases or disassociates licenses
  - preserves the individual/family record for traceability

### Employee

- employee disable must not release licenses
- employee purge requires inactive status first
- employee purge releases or disassociates licenses and preserves the employee record for traceability

### Hosted tenant authorization

- tenant authorization `disable` is a local suspension signal
- tenant authorization `enable` reactivates the local authorization state
- this is not the same as credential revocation
- this is not an implicit license release

## Communication note

In GW CORE, `Communication` is not the canonical execution mechanism for lifecycle commands.

For now:

- lifecycle commands should be modeled directly on the target resource
- `Communication` may later become a transport or evidence envelope for those commands
- future extension repositories may reuse the same lifecycle base and then wrap it in conversational or production-thread flows

## SDK guidance

SDK prompts should follow this order:

1. Read this document for stable lifecycle semantics.
2. Read `docs/90.L-LIFECYCLE_CURRENT_VS_TARGET.md` to distinguish current GW behavior from target normalization.
3. Read `CHANGELOG.md` to identify what has already shipped in the current version.

SDKs should prioritize the current deployed GW CORE contract, while leaving explicit TODOs for the target normalized `PATCH` model where the backend has not migrated yet.

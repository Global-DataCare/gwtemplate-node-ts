# Dataspace DID Services (DSP/DCP)

This document defines the DID `service` entries published for dataspace interoperability and the expected endpoint paths.

## Purpose

- Keep DSP/DCP discovery services explicit in tenant DID documents.
- Separate discovery/compliance service publication from internal API authorization templates.
- Provide a stable contract for conformance tests.

## Scope

This profile applies to **tenant DID documents** (providers/connectors).  
Host/ICA/Root may expose additional trust/compliance endpoints, but tenant publication is the baseline for integration.

## Catalog responsibility model

- **Tenant**: owns and publishes its local catalog entries (datasets/services/policies).
- **Host**: acts as broker/federator over tenant catalogs for discovery.
- Brokering metadata at host level does not imply host ownership of tenant clinical content.
- Tenant DID `service` publication remains the primary machine-readable
  discovery contract for DSP/DCP services.
- Additional host or tenant `well-known` discovery artifacts may be useful, but
  they complement rather than replace DID service publication.

## Published DID Services (Tenant)

The tenant DID document publishes these explicit entries:

1. `DataService`
   - `serviceEndpoint`: `/.well-known/dspace-version`
   - Purpose: DSP version discovery (`2025-1` profile and root path binding metadata).
2. `CatalogService`
   - `serviceEndpoint`: `/dcat3/catalog/request`
   - Purpose: project-local DSP catalog request entrypoint.
3. `IssuerService`
   - `serviceEndpoint`: `/presentations/query`
   - Purpose: DCP credential/presentation query discovery.

`/dcat3/catalog/request` is the current GW CORE runtime contract for tenant DID
service publication. It is a project-local binding that maps to the DSP catalog
request capability. The DSP standard HTTPS binding path is `/catalog/request`.

Host operator discovery catalogs are a separate publication surface and may be
exposed through a host DID `CatalogService` or a host discovery artifact such
as `/.well-known/dcat3/catalog`. Those host-level artifacts complement tenant
DID service publication; they do not replace the tenant `CatalogService`
contract documented here.

For hosted tenants, these paths are resolved under:

`/{tenantId}/cds-{jurisdiction}/{version}/{sector}`

Example (hosted tenant `acme`, `es`, `v1`, `health-care`):

- `https://gateway.example.com/acme/cds-es/v1/health-care/.well-known/dspace-version`
- `https://gateway.example.com/acme/cds-es/v1/health-care/dcat3/catalog/request`
- `https://gateway.example.com/acme/cds-es/v1/health-care/presentations/query`

## Path Bindings Matrix (Reference)

The following DSP/DCP path bindings are tracked as interoperability targets.
When GW CORE uses a local alias, both the DSP standard path and the current
project-local path are shown:

- DSP discovery: `/.well-known/dspace-version`
- DSP catalog:
  - Standard DSP:
    - `POST /catalog/request`
    - `GET /catalog/datasets/{id}`
  - Current GW CORE local binding:
    - `POST /dcat3/catalog/request`
    - `GET /dcat3/catalog/datasets/{id}`
- DSP negotiations:
  - `GET /negotiations/{providerPid}`
  - `POST /negotiations/request`
  - `POST /negotiations/{providerPid}/request`
  - `POST /negotiations/{providerPid}/events`
  - `POST /negotiations/{providerPid}/agreement/verification`
- DSP transfers:
  - `GET /transfers/{providerPid}`
  - `POST /transfers/request`
  - `POST /transfers/{providerPid}/start`
  - `POST /transfers/{providerPid}/completion`
  - `POST /transfers/{providerPid}/termination`
  - `POST /transfers/{providerPid}/suspension`
- DCP:
  - `POST /presentations/query`

Note: This document defines DID service publication and target path contracts.  
Endpoint runtime wiring is tracked independently from DID service publication.

## Compliance Test

Conformance is enforced by:

- `src/__tests__/unit/utils/dataspace.did-services.compliance.test.ts`

The test asserts that tenant DID service arrays include:

- `#dsp-data-service` (`DataService`)
- `#dsp-catalog-service` (`CatalogService`)
- `#dcp-issuer-service` (`IssuerService`)

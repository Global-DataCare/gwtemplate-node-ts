# Use Case cURL Examples

This document provides complete, copy-paste friendly `cURL` examples for common business use cases. These examples are designed for developers and integrators to test the API flow manually in a development environment.

## Prerequisites

1.  **Run the Server**: Start the server in development mode.
    ```bash
    npm run dev
    ```
2.  **Bearer Token**: For these tests, you will need a valid Bearer Token. In a local `dev` environment, you can use mock tokens configured in your `.env` file (e.g., `RECEPTIONIST1_ID_TOKEN`). For the examples below, replace `{YOUR_BEARER_TOKEN}` with a valid token.
3.  **Entity IDs**: You will need the `urn:uuid` identifiers for the entities involved (the customer/patient and the organization). These should be obtained from the system after the entities are created. Replace the placeholders like `{CUSTOMER_URN_UUID}` and `{ORGANIZATION_URN_UUID}` with these actual values.

## Flow: Secure Appointment Notification

This flow demonstrates how to establish an authorized communication channel and then send a notification through it.

### Step 1: Establish Communication Channel (Create Consent with Granular ODRL)

Before sending or querying data, the client MUST establish an authorized channel by creating a FHIR `Consent` resource. The `id` of this resource becomes the unique identifier for the channel.

This advanced example demonstrates a granular ODRL policy with three distinct permissions:
1.  Allow creating any `Communication`.
2.  Allow updating any `Composition`.
3.  Allow reading an `AppointmentResponse` **only if** the `Appointment` it refers to has an indexed attribute (`org.hl7.fhir.api.Appointment.serviceType`) that starts with a specific `did:web` prefix, effectively restricting access to a specific department or service.

The `leftOperand` in the ODRL constraint refers directly to the **name of an indexed attribute** in the confidential storage. When resources like `Appointment` or `AppointmentResponse` are saved, the system automatically creates indexed attributes for key fields. For example, when an `AppointmentResponse` is stored, two critical indices are created for its `appointment` reference:

-   `org.hl7.fhir.api.AppointmentResponse.appointment.reference.identifier.system`: Stores the service/location identifier (e.g., `did:web:hospital-a:department:cardiology...`). This allows for broad queries like "find all responses for the cardiology department".
-   `org.hl7.fhir.api.AppointmentResponse.appointment.reference.identifier.value`: Stores the specific UUID of the appointment (e.g., `urn:uuid:<uuid>`), which is hashed for security. This allows for direct lookups for a single appointment's response.

The ODRL policy below leverages the first, broader index to grant permission.

**Note:** For copy-pasting, the ODRL policy is provided here as a pre-encoded Base64 string. The placeholders inside the policy (`{...}`) would be populated by the server when the consent is created.

```bash

# - {YOUR_BEARER_TOKEN} with a valid token (e.g., from your .env file).
```

### Step 2: Send an Authorized FHIR Communication

With the granular channel established, an authorized employee sends a `Communication`. The authorization logic for *sending* remains the same: the `partOf.reference` must match the `Consent.id`. The ODRL policy within that consent will be enforced for any subsequent, more complex queries (like a professional reading the patient's `AppointmentResponse`).

```bash
# MANUALLY REPLACE:
# - {YOUR_BEARER_TOKEN}, {CUSTOMER_URN_UUID}, {ORGANIZATION_URN_UUID} as before.
# - The CHANNEL_ID in "partOf.reference" must match the Consent ID from Step 1.

curl -i --location --request POST 'http://localhost:3000/acme/cds-ES/v1/health-care/individual/org.hl7.fhir.r4/Communication' \
--header "Authorization: Bearer {YOUR_BEARER_TOKEN}" \
--header 'Content-Type: application/json' \
--data-raw '{
```

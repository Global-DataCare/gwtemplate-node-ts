# Real-time API Test using `curl`

This guide provides `curl` commands to simulate the API requests from the integration tests, allowing for direct observation of the server's behavior and logs.

---

## Step 1: Configure Your Environment

All commands in this guide use environment variables for the server URL and authentication tokens. This makes it easy to switch between local and deployed environments.

### a. Set the Base URL

Open your terminal and export the `BASE_URL` variable.

**For local development:**
```bash
export BASE_URL="http://localhost:3000"
```

**For a deployed Cloud Run environment:**
*(Replace the URL with your actual service URL)*
```bash
export BASE_URL="https://your-service-name.run.app"
```

### b. Set Authentication Tokens

For `demo` mode, the server decodes JWTs but does not verify their signature. Export the following variables in your terminal.

```bash
# Tokens with "alg": "none" for NODE_ENV=demo.
export ADMIN1_ID_TOKEN="eyJhbGciOiJub25lIn0.eyJlbWFpbCI6ImFkbWluMUBhY21lLm9yZyIsIm5hbWUiOiJEZW1vIEFkbWluIn0."
export RECEPTIONIST1_ID_TOKEN="eyJhbGciOiJub25lIn0.eyJlbWFpbCI6InJlY2VwdGlvbmlzdDFAYXBpLmFjbWUub3JnIiwibmFtZSI6IkRlbW8gUmVjZXB0aW9uaXN0In0."
```
**Tip:** Add these `export` commands to your `~/.bash_profile` or `~/.zshrc` file to make them permanent.

---

## Step 2: A Note on `jq` and Debugging

The `curl` commands in this guide **do not** pipe the output to `jq` by default. This is intentional.

If the server returns an error (e.g., an HTML error page or plain text), `jq` would fail with a JSON parsing error, hiding the original, useful error message from the server.

**Recommendation:**
1.  Run the commands as they are written first.
2.  Once you confirm you are getting a successful JSON response, you can append `| jq '.'` to the end of the command to format the output for better readability.

### 7. Establish a Communication Channel (Create Consent with ODRL)

This command creates a FHIR `Consent` resource, which acts as an authorized channel for future communications.

```bash
# Replace {CUSTOMER_URN_UUID} and {ORGANIZATION_URN_UUID} with the actual IDs from previous steps.
CHANNEL_ID="urn:uuid:channel-12345"
THID="thid-consent-$(date +%s)"
ODRL_POLICY='{ "@context": "http://www.w3.org/ns/odrl.jsonld", "@type": "Agreement", "uid": "urn:uuid:odrl-policy-123", "target": "urn:uuid:{CUSTOMER_URN_UUID}", "assigner": "urn:uuid:{CUSTOMER_URN_UUID}", "assignee": "urn:uuid:{ORGANIZATION_URN_UUID}", "permission": [{ "action": "read", "target": "https://connecthealth.com/fhir/Communication", "constraint": [{ "leftOperand": "purpose", "operator": "eq", "rightOperand": "TREAT" }] }] }'
ODRL_BASE64=$(echo -n "$ODRL_POLICY" | base64)

curl -i --location --request POST "${BASE_URL}/acme/cds-ES/v1/health-care/individual/org.hl7.fhir.r4/Consent" \
--header "Authorization: Bearer $RECEPTIONIST1_ID_TOKEN" \
--header 'Content-Type: application/json' \
--data-raw '{
    "thid": "'"$THID"'",
    "iss": "did:web:api.acme.org:employee:email:receptionist1@api.acme.org:role:ISCO-08:4226",
    "aud": "urn:antifraud:health-care:acme",
    "body": { "resourceType": "Consent", "id": "'"$CHANNEL_ID"'", "status": "active", "scope": { "coding": [{ "system": "http://terminology.hl7.org/CodeSystem/consentscope", "code": "patient-privacy" }] }, "category": [{ "coding": [{ "system": "http://terminology.hl7.org/CodeSystem/consentcategorycodes", "code": "comm" }] }], "patient": { "reference": "urn:uuid:{CUSTOMER_URN_UUID}" }, "performer": [{ "reference": "urn:uuid:{ORGANIZATION_URN_UUID}" }], "sourceAttachment": { "contentType": "application/odrl+json", "data": "'"$ODRL_BASE64"'" }, "provision": { "type": "permit" } }
}'
```

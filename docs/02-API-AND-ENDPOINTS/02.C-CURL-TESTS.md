# Real-time API Test using `curl`

This guide provides the steps to start the server in `demo` mode and use `curl` to simulate the API requests from the integration test. This allows for direct observation of the server's behavior and debug logs without the complexity of the Jest test runner.

## Step 1: Set Up Environment Variables

This testing method relies on demo JWTs (Bearer Tokens) defined as environment variables. In `demo` mode, the server decodes these tokens to read their `email` claim but **does not verify the cryptographic signature**.

Ensure the following variables are present in your `.env` file. They contain JWTs with `alg=none` and the appropriate email claim for each test user.

```bash
# --- Demo JWT Tokens for cURL Tests ---
# Tokens with "alg": "none" for NODE_ENV=demo.
ADMIN1_ID_TOKEN="eyJhbGciOiJub25lIn0.eyJlbWFpbCI6ImFkbWluMUBhY21lLm9yZyIsIm5hbWUiOiJEZW1vIEFkbWluIn0."
RECEPTIONIST1_ID_TOKEN="eyJhbGciOiJub25lIn0.eyJlbWFpbCI6InJlY2VwdGlvbmlzdDFAYXBpLmFjbWUub3JnIiwibmFtZSI6IkRlbW8gUmVjZXB0aW9uaXN0In0."
```

---

## Step 2: Run the Server in Demo Mode

First, start the API server. The `dev` script uses `ts-node-dev` and loads the environment variables from the `.env` file, which should set `NODE_ENV=demo` and `PORT=3000`.

**Important:** If you just added the token variables, you must restart the server for them to be loaded.

In your project's root directory, open a terminal and run the following command. Leave this terminal running to monitor the server's log output.

```bash
npm run dev
```

You should see the server's startup logs, ending with a message similar to this:

```
[GW-API] demo Server running on http://localhost:3000
[GW-API] --- System Initialized Successfully ---
```

## Step 3: End-to-End Test Flow

Open a **new, separate terminal** to run the following `curl` commands. This flow simulates the entire process of registering an organization, adding an employee, and onboarding a customer.

### 3.1. Create the 'acme' Organization

This command sends the organization registration payload to the `host`'s registry endpoint, authenticated as `admin1@acme.org`. Use `curl -i` to see the `location` header in the response.

```bash
curl -i --location --request POST 'http://localhost:3000/host/cds-ES/v1/test/registry/org.schema/Organization/_batch' \
--header "Authorization: Bearer $ADMIN1_ID_TOKEN" \
--header 'Content-Type: application/json' \
--data-raw '{
  "thid": "thid-org-c1c2c3d4-e5f6-7890-1234-567890abcdef",
  "iss": "admin1@acme.org",
  "aud": "did:web:host.example.com",
  "body": { "data": [ { "type": "Organization-registration-form-v1.0", "meta": { "claims": { "@context": "org.schema", "@type": "template", "org.schema.Organization.legalName": "Acme Organization", "org.schema.Organization.identifier.additionalType": "TAX", "org.schema.Organization.identifier.value": "A123456789", "org.schema.Organization.alternateName": "acme", "org.schema.Organization.address.addressCountry": "ES", "org.schema.Person.identifier": "urn:uuid:b1b2c3d4-e5f6-7890-1234-567890abcdef", "org.schema.Person.hasOccupation": "ISCO-08:1120", "org.schema.Person.email": "admin1@acme.org", "org.schema.Service.category": "health-care", "org.schema.Service.identifier": "urn:uuid:d1c2c3d4-e5f6-7890-1234-567890abcdef", "org.schema.Service.termsOfService": "https://provider.example.com/terms", "org.schema.Service.serviceType": "http://terminology.hl7.org/CodeSystem/v3-ActReason|SRVC" } } } ] }
}'
```

### 3.2. Poll for Organization Creation Result

After receiving the `202 Accepted` response, use the `location` header from the previous step to poll for the result. The polling endpoint expects a `POST` request with the `thid` in the JSON body.

```bash
# The URL is provided in the 'location' header of the previous response.
# It should end in /_batch-response
curl -i --location --request POST 'http://localhost:3000/host/cds-ES/v1/test/registry/org.schema/Organization/_batch-response' \
--header 'Content-Type: application/json' \
--data-raw '{
    "thid": "thid-org-c1c2c3d4-e5f6-7890-1234-567890abcdef"
}' | jq '.'
```
**Expected Outcome:** A `200 OK` response with a JSON payload confirming the creation of the resources.

For this final step, you can use manual polling or the automated script.

#### Option A: Manual Polling
```bash

curl --location --request POST 'http://localhost:3000/acme/cds-es/v1/health-care/individual/org.schema/Person/_batch-response' \
```

#### Option B: Automated Polling Script

Save this script as `test_customer_creation.sh` and run it with `./test_customer_creation.sh`.

```bash
#!/bin/bash

# --- Step 1: Execute the initial POST request to create the customer ---
echo "INFO: Sending the request to create a new customer..."

# We store the response headers in a temporary file
HEADERS_FILE=$(mktemp)
THID="thid-e2e-person-onboarding-cli-$(date +%s)" # Append timestamp to make thid unique

# Execute curl:
# -s: Silent mode (no progress meter)
# -D $HEADERS_FILE: Dump response headers to the file
# -o /dev/null: Discard the response body (which is empty on a 202)
curl -s \
  -D $HEADERS_FILE \
  -o /dev/null \
  --location --request POST 'http://localhost:3000/acme/cds-es/v1/health-care/individual/org.schema/Person/_batch' \
  --header "Authorization: Bearer $RECEPTIONIST1_ID_TOKEN" \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "thid": "'"$THID"'",
    "iss": "did:web:api.acme.org:employee:email:receptionist1@api.acme.org:role:ISCO-08:4226",
    "aud": "urn:antifraud:health-care:acme",
    "body": { "data": [ { "meta": { "claims": { "org.schema.Person.alternateName": "Joe", "org.schema.Person.identifier": "urn:uuid:8e0d846a-2492-4b9c-8a4e-5e065fb6ba76", "org.schema.Person.email": "customer1@example.com", "org.schema.Service.category": "health-care", "org.schema.Service.termsOfService": "https://provider.example.com/terms", "org.schema.Service.serviceType": "http://terminology.hl7.org/CodeSystem/v3-ActReason|FAMRQT,PWATRNY,METAMGT,FRAUD,RECORDMGT,COVAUTH,TREAT,DISASTER,HPAYMT,MLTRAINING,ETREAT,HOPERAT,CAREMGT,HSYSADMIN,PATADMIN,PATSFTY" } }, "request": { "method": "POST", "url": "individual/org.schema/Person/" }, "type": "Individual-terms-v1.0" }, { "meta": { "claims": { "org.schema.Person.identifier": "urn:uuid:8e0d846a-2492-4b9c-8a4e-5e065fb6ba76", "org.schema.Person.identifierType": "NNES", "org.schema.Person.identifierValue": "12345678X" } }, "request": { "method": "POST", "url": "individual/org.schema/Person/" }, "type": "Personal-identity-v1.0" } ] }
  }'

# --- Step 2: Extract the polling URL from the headers ---
# The location header contains just the path, so we prepend the base URL
LOCATION_PATH=$(grep -i '^location:' $HEADERS_FILE | awk '{print $2}' | tr -d '\r')

if [ -z "$LOCATION_PATH" ]; then
  echo "ERROR: Could not find 'location' header in the response."
  cat $HEADERS_FILE
  rm $HEADERS_FILE
  exit 1
fi

POLLING_URL="http://localhost:3000${LOCATION_PATH}"
echo "INFO: Request accepted. Polling URL: $POLLING_URL"

# --- Step 3: Poll the endpoint until the job is complete ---
STATUS_CODE="000"
while [ "$STATUS_CODE" != "200" ]; do
  echo "INFO: Polling job status... (waiting 2s)"
  sleep 2
  # We run curl to get only the HTTP status code, using a POST request
  # with the thid in the body.
  STATUS_CODE=$(curl -s -w '%{http_code}' -o /dev/null --location --request POST "$POLLING_URL" \
    --header 'Content-Type: application/json' \
    --data-raw '{"thid": "'"$THID"'"}')
done

echo "INFO: Job complete! Status code 200 received."

# --- Step 4: Fetch and display the final result ---
echo "INFO: Fetching the final result..."
# Use jq to format the JSON output
curl -s --location --request POST "$POLLING_URL" \
  --header 'Content-Type: application/json' \
  --data-raw '{"thid": "'"$THID"'"}' | jq '.'

# Clean up the temporary file
rm $HEADERS_FILE
```

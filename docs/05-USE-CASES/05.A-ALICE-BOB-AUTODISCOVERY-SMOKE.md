# Alice/Bob Autodiscovery Smoke

## Purpose

This runbook validates the current service autodiscovery surface with two
independent GW hosts:

- `Alice`: one provider tenant
- `Bob`: multiple tenants, including both provider and reader-only cases

It exercises only the current service autodiscovery contract:

- `GET /.well-known/dcat3/catalog`
- `POST /api/dataspace-discovery/providers`

This runbook does not cover tenant dataset discovery inside a
`DigitalTwinProvider`. That is a separate track.

## Local Files

Versioned templates:

- `env.alice.example`
- `env.bob.example`

Local private copies expected by the scripts:

```bash
cp env.alice.example .env.alice
cp env.bob.example .env.bob
```

The repo ignores `.env.*`, so the committed source of truth remains the visible
`env.*.example` files.

## Host Topology

### Alice

- base URL: `http://localhost:3000`
- host role: local node operator
- published tenant:
  - `Acme 1` with `indexing.cruds`

### Bob

- base URL: `http://localhost:3001`
- host role: local node operator
- published tenants:
  - `Acme 2` with `indexing.cruds`
  - `Acme 3` with `digitaltwin.cruds`
- non-published tenant:
  - `Acme 4` with `indexing.rs`

The reader-only tenant exists to prove that host catalogs and normalized
provider discovery exclude non-provider capabilities.

## Flow Context

The operational route order stays:

1. Host or tenant activation material is accepted by GW.
2. Initial access token exchange happens at `Token/_exchange`.
3. Technical device/app registration happens at `Device/_dcr`.
4. If ICA-backed X.509 issuance is required, the tenant/host submits a CSR
   through GW and GW calls ICA internally.
5. Only operational provider-capable tenants are published through host
   autodiscovery.

Autodiscovery therefore validates publication after onboarding, not the full
identity/bootstrap flow itself.

## Start Both Hosts

```bash
bash ./scripts/run-alice-bob-local.sh
```

Optional overrides:

```bash
ALICE_ENV=.env.alice \
BOB_ENV=.env.bob \
ALICE_BASE_URL=http://localhost:3000 \
BOB_BASE_URL=http://localhost:3001 \
bash ./scripts/run-alice-bob-local.sh
```

The launcher:

- closes local listeners on ports `3000` and `3001`
- runs `npm run build:swagger` once
- starts both GW instances in the background
- writes logs into `logs/alice-gw.log` and `logs/bob-gw.log`
- waits until `GET /host/.well-known/ping` returns `200`

## Bootstrap Tenant Matrix

```bash
bash ./scripts/bootstrap-alice-bob-discovery.sh
```

This uses the existing `bootstrap-single-tenant.sh` flow, but now also sets:

- `SERVICE_URL`
- `SERVICE_AREA_SERVED`
- `SERVICE_TYPE`

That matters because the host autodiscovery catalog publishes only tenants whose
persisted `org.schema.Service.serviceType` denotes a provider role.

## Run The Smoke

```bash
bash ./scripts/smoke-alice-bob-autodiscovery.sh
```

Checks performed:

1. Alice host catalog publishes exactly one provider dataset.
2. Bob host catalog publishes exactly two provider datasets.
3. Bob host catalog excludes the reader-only tenant.
4. Alice normalized discovery resolves exactly one index provider.
5. Bob normalized discovery resolves two providers without a capability filter.
6. Bob filtered discovery resolves one `indexing.cruds` provider.
7. Bob filtered discovery resolves one `digitaltwin.cruds` provider.

## Stop The Local Hosts

```bash
PORTS=3000,3001 bash ./scripts/local-close.sh
```

## Expected Outcome

If the smoke passes, the current GW CORE autodiscovery surface proves that:

- host publication is runtime-owned by GW
- publication is derived from operational provider-capable tenants
- reader-only tenants are not exposed publicly
- two different node operators can publish overlapping service capabilities in
  the same sector without collapsing into a single-host assumption

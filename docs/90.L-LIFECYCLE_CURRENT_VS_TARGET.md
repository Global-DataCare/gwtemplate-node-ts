## Lifecycle Current Vs Target

Status: Transitional.

This document separates the current GW CORE implementation from the target lifecycle contract discussed for SDK and shared-package alignment.

Use this document to avoid implementing target semantics prematurely in SDKs or downstream repos.

For the stable local lifecycle rules, read [01.I-LIFECYCLE-101.md](01-OVERVIEW-AND-GUIDES/01.I-LIFECYCLE-101.md) first.

## Current GW CORE

### Current individual/family contract

- `individual/org.schema/Organization/_batch`
  - current onboarding/create path
- `individual/org.schema/Organization/_transaction`
  - current alias for the onboarding path
- `individual/org.schema/Organization/_search`
  - search path
- `individual/org.schema/Organization/_disable`
  - explicit lifecycle action
- `individual/org.schema/Organization/_purge`
  - explicit lifecycle action

Current behavior:

- disable marks the registration inactive
- disable does not release licenses
- purge requires prior inactive status
- purge releases or disassociates licenses
- purge preserves the record for traceability

### Current employee contract

- `entity/org.schema/Employee/_batch`
  - current create path
  - current disable path is still modeled via `DELETE` on entry method inside the batch flow
- `entity/org.schema/Employee/_purge`
  - explicit lifecycle action

Current behavior:

- employee disable does not release licenses
- employee purge requires prior inactive status
- employee purge releases or disassociates licenses
- employee purge preserves the employee record

### Current host tenant authorization contract

- `registry/org.schema/Organization/_activate`
- `registry/org.schema/Organization/_disable`
- `registry/org.schema/Organization/_enable`

Current behavior:

- this is local GW tenant authorization lifecycle
- it is not credential revocation
- it is not automatic license release

## Target normalized contract

The target model is transport-neutral but should be expressible consistently in GW, shared utilities, and SDKs.

### Target action rules

- `_search` for search semantics
- `_batch + POST` for create
- `_batch + PUT` for resource data update
- `_batch + PATCH` for lifecycle or partial operational updates
- `/_purge + POST` for explicit purge semantics
- `DELETE` reserved for true delete semantics only if that domain really needs them

### Target storage rules

- `ConfidentialStorageDoc.status` remains the canonical current status
- `audit` stores transition metadata
- blockchain receives a derived status-change event

Recommended audit extension:

- `statusChangedAt`
- `statusChangedBy`
- `previousStatus`
- `statusReason`

## Migration intent

The intended migration path is:

1. Keep current explicit `_disable` and `/_purge` routes working.
2. Document target `PATCH` semantics clearly.
3. Extend shared models and SDKs around the normalized contract.
4. Only then migrate GW manager internals and route contracts where appropriate.

## SDK prompt rule

When generating or updating SDK code:

- implement the current GW CORE contract first
- use the target contract as the forward design
- leave explicit TODOs where current GW behavior is expected to migrate toward `_batch + PATCH`

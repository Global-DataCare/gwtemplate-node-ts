# DID Endpoint Resolution Task

Purpose: executable instructions for a separate implementation/review thread.

This task clarifies how CORE GW should publish:

- public-facing `did:web` identifiers
- visible aliases such as `globaldatacare.es`
- real invocable `serviceEndpoint` URLs from node operators (e.g. `accuro.es`, `unid.online` or IPs).

The main rule to preserve:

- `did:web` is the public identity
- `alsoKnownAs` contains aliases
- `service[].serviceEndpoint` must contain the real callable URL the SDK/app should use

This task must also preserve a strict key-separation rule:

- transport/device keys are not the same as person-signing keys
- portal/app/native device communication keys are not the same as controller/professional/individual/related-person signing keys

Do not make SDKs reconstruct API URLs by concatenating `did:web` parts with guessed GW paths.

## Concrete Use Case

Example public organization DID:

```text
did:web:globaldatacare.es:health-care:organization:taxid:VATES-B00112233
```

Example real node operator endpoint:

```text
https://<node.operator>/VATES-B00112233/cds-ES/v1/health-care/identity/openid/smart/token
```

Possible public visible alias:

```text
https://globaldatacare.es/health-care/organization/taxid/VATES-B00112233
```

Desired behavior:

1. A portal/app resolves the public DID.
2. The DID Document is returned from the public domain.
3. The DID Document exposes the real `serviceEndpoint` values to call.
4. The SDK calls the `serviceEndpoint` directly.

Important:

- `alsoKnownAs` is for aliases, branding, and alternative visible locations.
- `serviceEndpoint` is for the actual API URL.
- The SDK must not need to combine `did:web` plus path pieces to invent the backend URL.

## Mandatory Key Separation

This thread must not mix these two identity layers:

1. Device / portal / app transport identity
- used for DIDComm/JOSE transport
- used by the portal web or native app as the technical runtime identity
- contains communication keys such as:
  - `comm_sig`
  - encryption / key agreement keys
- this is the identity bound in DCR-style device registration flows

2. Person / actor signing identity
- used by the real actor:
  - controller
  - professional
  - individual
  - related person
- used for person-level authorization and proof-bearing operations
- especially relevant when requesting SMART access tokens or signing human-controlled actions

Operational rule:

- DCR is for the device/app/portal runtime identity
- SMART access token requests are about the person/actor identity
- the SDK and GW must keep both identities available and distinguishable

## Practical Interpretation For CORE GW

When a tenant/individual service is activated:

- the GW must be able to publish a DID Document for the tenant/provider identity
- that DID Document must expose the transport/public keys managed by the connector/GW for communications
- if the controller's public JWK set or signing `kid` is needed to bind the initial human controller to the service, that must be carried explicitly by the activation flow

Do not assume:

- the controller signing keys and the transport keys are interchangeable
- the person DID document and the device/portal DID document are the same thing

## Target DID Document Model

Illustrative shape:

```json
{
  "@context": "https://www.w3.org/ns/did/v1",
  "id": "did:web:globaldatacare.es:health-care:organization:taxid:VATES-B00112233",
  "alsoKnownAs": [
    "https://globaldatacare.es/health-care/organization/taxid/VATES-B00112233",
    "https://<node.operator>/VATES-B00112233/cds-ES/v1/health-care/"
  ],
  "service": [
    {
      "id": "#did-document",
      "type": "LinkedDomains",
      "serviceEndpoint": "https://globaldatacare.es/health-care/organization/taxid/VATES-B00112233/.well-known/did.json"
    },
    {
      "id": "#jwks",
      "type": "JsonWebKeyService2020",
      "serviceEndpoint": "https://globaldatacare.es/health-care/organization/taxid/VATES-B00112233/jwks.json"
    },
    {
      "id": "#identity-openid-smart-token",
      "type": "ApiService",
      "serviceEndpoint": "https://<node.operator>/VATES-B00112233/cds-ES/v1/health-care/identity/openid/smart/token"
    }
  ]
}
```

This means:

- public DID resolution can stay under `globaldatacare.es`
- actual operational traffic can go to `<node.operator>`

## Individual and Employee DIDs

The same identity-vs-endpoint rule applies to:

- employee/professional DIDs
- individual DIDs
- `RelatedPerson` DIDs

Recommended resolution split:

- actor DID Document:
  - actor public keys
  - actor aliases
- organization/provider DID Document:
  - tenant/provider public keys
  - callable service endpoints

The SDK should normally resolve service endpoints from the organization/provider DID Document, and actor keys from the actor DID Document.

## Recommended Design Decision

Default recommendation:

- keep public identity stable and human-facing
- publish real operational URLs in `serviceEndpoint`
- use `alsoKnownAs` for aliases only
- prefer direct invocable URLs over pretty URLs in `serviceEndpoint`
- keep device transport keys and person-signing keys explicitly separated in code, docs, and DID publication logic

That means:

- if `<node.operator>` is the real stable endpoint, publish `<node.operator>`
- if `globaldatacare.es` is a stable reverse proxy and you want traffic to enter there, publish `globaldatacare.es`

But do not require the SDK to guess or reconstruct this.

## Acceptance Criteria

The task is done when all of this is true:

1. Resolving a public `did:web` returns a DID Document with real callable URLs.
2. `serviceEndpoint` URLs are directly invocable without client-side reconstruction.
3. `alsoKnownAs` is used only for aliases, not as a routing substitute.
4. Documentation clearly distinguishes:
   - public DID
   - alias domain
   - operational API endpoint
   - device/portal transport identity
   - person/actor signing identity
5. SDK examples can say:
   - resolve DID
   - read `serviceEndpoint`
   - call it directly
6. The task explains clearly whether DCR is for the device identity, the person identity, or both in separate phases.

## Important Non-Goals

This task is not about:

- redesigning all individual/professional DID shapes
- changing SMART scope semantics
- changing consent semantics
- changing JWKS key purposes
- collapsing device and person identities into a single key model

It is specifically about public DID resolution versus real service endpoint publication.

Title: ICA Certificate Issuance + Ledger Registration (Test Network)

Status: Draft

Purpose
This guide explains how a Host/Operator and Tenant obtain an ICA‑signed X.509
certificate and how the gateway registers the organization on Hyperledger Fabric
with signed PDF evidence.

Scope
- Test network onboarding (`test-network`).
- Host and Tenant flows.
- Evidence is a signed PDF (PAdES) of Terms & Conditions.

## Flow Position

This document covers only the CSR and ICA issuance slice of the broader GW
onboarding/runtime lifecycle.

The larger route order is:

1. host operator or tenant onboarding reaches GW
2. tenant activation is completed in GW
3. `Token/_exchange` grants initial technical access
4. `Device/_dcr` registers the technical client identity
5. CSR enrollment through ICA signs the runtime certificate when needed
6. host autodiscovery publishes only operational provider-capable tenants

`Token/_exchange` and `Device/_dcr` are not certificate issuance steps. They
bootstrap the app/device technical identity used to call the tenant runtime.
CSR issuance is a parallel trust artifact for the host or tenant service
endpoint itself.

1) Key + CSR generation (local)
- The private key is generated locally inside the gateway/KMS. It is NEVER uploaded.
- A CSR is built locally from that key pair.
- The CN for the cert uses the service name (alternateName), not the legal name.

2) ICA issuance (external)
- The gateway submits the CSR + onboarding evidence to the ICA.
- ICA verifies:
  - Terms & Conditions signature (PDF signature validation).
  - Organization metadata (schema.org Organization).
  - Legal representative identity.
- ICA returns an X.509 leaf certificate plus chain.

Publication rule:

- ICA issuing a certificate does not by itself make the tenant publicly
  discoverable.
- Public discovery remains a host-side GW publication step derived from the
  tenant operational state and service capabilities.

ICA Endpoint (gwtemplate)
Path (host‑only):
```
POST /host/cds-<jurisdiction>/v1/system/test-network/ica/csr/_enroll
```
Routing mapping:
- `sector=system`
- `section=test-network`
- `format=ica`
- `resourceType=csr`
- `action=_enroll`

Request body (minimal):
```
{
  "csr": "-----BEGIN CERTIFICATE REQUEST-----...-----END CERTIFICATE REQUEST-----",
  "organization": { ...schema.org Organization... },
  "evidence": [ ...DocumentSignature evidence... ],
  "metadata": { "environment": "test-network" }
}
```

Response (auto‑approve dev mode):
```
{
  "status": "approved",
  "certificate": "-----BEGIN CERTIFICATE-----...",
  "chain": ["-----BEGIN CERTIFICATE-----..."],
  "caName": "ica"
}
```

ICA dev env (auto‑approve):
- `ICA_AUTO_APPROVE=true`
- `ICA_FABRIC_CA_ENROLL_URL=https://<fabric-ca-host>/api/v1/enroll`
- `ICA_FABRIC_CA_AUTH=Basic <base64(enrollId:enrollSecret)>`
- `ICA_FABRIC_CA_NAME=ica` (optional)
- `ICA_FABRIC_CA_PROFILE=` (optional)
- `ICA_FABRIC_CA_LABEL=` (optional)

Mutual TLS (recommended for internal ICA):
- `ICA_MTLS_CERT_PEM` or `ICA_MTLS_CERT_FILE`
- `ICA_MTLS_KEY_PEM` or `ICA_MTLS_KEY_FILE`
- `ICA_MTLS_CA_PEM` or `ICA_MTLS_CA_FILE` (optional)

mTLS key reuse (current default)
- The client mTLS certificate/key **may reuse** the organization X.509 leaf cert/key.
- This is acceptable for now and simplifies bootstrap.
- Future hardening can split mTLS into a dedicated client cert (separate keypair + rotation policy).

Current source for mTLS material (runtime)
- First: vault `host` → section `pki` → id `ica-mtls` (certPem/keyPem)
- Fallback: legacy X.509 leaf cert from `EntityConfig.legacyX509DerBase64`
  + legacy ES384 private key from KMS (`getLegacyPrivateKeyPem('host')`)

Demo tenant slugs (local ICA/CA)
- In `NODE_ENV=demo`, if `ICA_EXTERNAL_DOMAIN` / `CA_EXTERNAL_DOMAIN` are set, the server
  creates reserved tenants whose `alternateName` is the domain slug:
  - `test-eur-ica.unid.online` → `test-eur-ica-unid-online`
  - `test-eur-root.unid.online` → `test-eur-root-unid-online`
- These tenants expose the ICA endpoint under:
  `/test-eur-ica-unid-online/cds-es/v1/system/test-network/ica/csr/_enroll`

Messaging endpoints (DIDComm v2)
Base path:
`/<tenant_id>/cds-<jurisdiction>/v1/<sector>/messaging/<format>/<resourceType>/<action>`

Example (post-quantum, send):
`/host/cds-es/v1/system/messaging/post-quantum/didcomm-encrypted/_send`

SDK guidance
- Preferred: applications perform DIDComm encrypt/sign/verify **locally** via SDK.
- Server endpoints exist as a **fallback** for systems without SDK support.

3) Gateway publishes cert (public)
The gateway exposes:
- `/.well-known/jwks.json` with `x5c` (and optionally `x5u`)
- `/.well-known/x509.der` (certificate chain)

For hosted tenants this publication is tenant-scoped. The host-level service
autodiscovery surface remains separate:

- `GET /.well-known/dcat3/catalog`
- `POST /api/dataspace-discovery/providers`

Those endpoints publish provider availability, not certificate issuance state.

4) Ledger registration (organization-sc)
When `LEDGER_ENABLED=true`, the gateway registers the org on the ledger after
successful onboarding (Order confirmation).

Environment
- `LEDGER_ENABLED=true`
- `LEDGER_MSP_ID=UNIDMSP` (or the MSP that submits the transaction)
- `LEDGER_ORG_CHAINCODE=organization-sc` (optional override)
- `GOVERNANCE_SCHEMA_URL=https://governance.example.org/schema/organization.jsonld`

Channel selection
- Identity channel is derived by jurisdiction: `eu-identity`, `na-identity`, etc.

Payload fields (organization-sc)
- `orgId`: organization identifier (URN)
- `schemaUrl`: governance schema URL
- `governanceVc`: legal-participant VC
- `selfDescriptionVc`: self-description VC
- `keys`: DID verificationMethod array (public keys)
- `evidence`: PDF signature evidence (see below)
- `metadata`: { role, sector, namespace, host, organization }

Evidence format (minimum)
The gateway extracts the PDF signature and stores the evidence as:
```
{
  "type": "DocumentSignature",
  "signature": {
    "type": "PAdES",
    "signatureValue": "<base64url-signature-bytes>"
  },
  "digest": [
    { "type": "DocumentHash", "hashAlg": "SHA256", "hashValue": "<base64url>" },
    { "type": "SignedDocumentHash", "hashAlg": "SHA256", "hashValue": "<base64url>" }
  ],
  "document": [
    { "type": "DigitalDocument", "hashAlg": "SHA256", "hashValue": "<base64url>" },
    { "type": "DigitalDocument", "hashAlg": "SHA256", "hashValue": "<base64url>" }
  ],
  "x5c": [ "<base64-der-cert>", "..." ]
}
```

Ledger conflict rule
If the same `(DocumentHash, SignedDocumentHash)` pair already exists in the
ledger, `organization-sc` returns `EvidenceAlreadyRegistered`, and the gateway
responds with HTTP 409.

Where this runs in code
- PDF evidence extraction: `src/utils/pdf-evidence.ts`
- Onboarding flow:
  - `HostingManager.processRegistrationEntry` stores evidence in the Service meta.
  - `HostingManager.processOrderEntry` registers the org on ledger.
  - `HostingManager.persistHostConfig` registers the host when enabled.

Notes
- For demo mode (`NODE_ENV=demo` or `NODE_ENV=test`), ledger registration is
  disabled unless `LEDGER_ENABLED=true`.
- The ICA issuance endpoint is still a TODO; implement it in the ICA service and
  wire the gateway to call it with CSR + evidence.
- Lifecycle TODO for `v1`:
  - tenant/employee/individual `disable` should be modeled as suspension, not deletion and not automatic license release.
  - authoritative credential suspension/revocation must come from ICA as a VC/ledger status update.
  - the host portal/GW should later support a canonical `/_deactivate` endpoint that ingests ICA-backed status updates and persists the corresponding suspended local state.

## Current Contract Summary

Use this separation when documenting or implementing the runtime:

- GW onboarding and activation make the tenant operational.
- `Token/_exchange` and `Device/_dcr` bind the technical client.
- ICA CSR issuance signs the service certificate.
- Host autodiscovery decides whether the tenant is publicly published.

That keeps CSR issuance, device registration, and service autodiscovery from
collapsing into one undocumented step.

# Architecture: Entity Key Management Lifecycle

This document outlines the architecture for entity cryptographic identity, covering the two primary onboarding flows (BYOK and Managed) and the subsequent lifecycle events like adding devices (DCR) and key rotation.

## 1. Core Principles

- **Two Onboarding Paths**: A new entity (specifically a Tenant Admin during organization registration) can either provide their own keys (BYOK) or have the system generate them (Managed).
- **Managed is the Default for Subsequent Entities**: When an admin creates subsequent entities like Employees, their keys are `Managed` by default. These entities can then add their own devices via a DCR flow.
- **BYOK is an Explicit Action (`_dcr`) for Existing Entities**: An existing entity wishing to add a new device with its own keys uses the DCR flow.
- **The DID Document is the Source of Truth**: An entity's `didDocument` is the single source of truth for their active public keys.

## 2. Onboarding Flows for Tenant Admins

### 2.1. Flow A: Bring-Your-Own-Key (BYOK) Onboarding

This is the recommended flow for Tenant Admins who have a secure wallet or key generation mechanism.

1.  **Request**: The prospective admin submits a `_batch` request to the `/host/.../registry` endpoint.
2.  **Requirement**: The JWE and JWS headers **MUST** contain the `jwk` parameter with the admin's public keys for encryption and signing.
3.  **System Action**:
    - The `HostingManager` verifies the request.
    - It extracts the provided `jwk`s from the headers.
    - It creates the `didDocument` for the new admin, using these `jwk`s to create the initial `verificationMethod` entries.
    - **No keys are generated by the server.** The admin's identity is immediately tied to the keys they exclusively control.

### 2.2. Flow B: Managed Key Onboarding

This is the fallback for admins who do not provide their own keys.

1.  **Request**: The prospective admin submits a request (either a JWE-protected request or a legacy JSON request with a Bearer token). The request **does not** contain the `jwk` parameter in the headers.
2.  **System Action**:
    - The system interprets the absence of `jwk` as a request for managed keys.
    - `HostingManager` calls `IKmsService.provisionKeys()` for the new admin entity.
    - The KMS generates a new deterministic key pair.
    - The `didDocument` for the new admin is created with the public keys of this new `managed` key pair.
3.  **User Action (Post-Onboarding)**: To use their identity, the user must interact with the system via the **Legacy Flow**, where they use their portion of the key `seed` to authorize the server to perform crypto operations on their behalf. See `03.G-LEGACY-AND-MANAGED-KEYS.md`.

## 3. Post-Onboarding: Adding New Devices (DCR)

This flow is for any existing entity (Admin, Employee) to register a new device/key.

1.  **Prerequisite**: The entity exists and has at least one active key.
2.  **Request**: The entity submits a request to a dedicated DCR endpoint (e.g., `.../device/_dcr`), signed with an existing active key.
3.  **Payload**: The request body contains the `JWKSet` of the new device and optional metadata (OS, push token).
4.  **System Action**:
    - The system verifies the signature with an existing key.
    - It adds the new public keys as `verificationMethod` entries to the entity's `didDocument`.
    - It stores device metadata in an encrypted `devices` array in the entity's configuration.
    - It updates the HMAC-indexed attributes to include the `kid`s of the new keys.

## 3. DID Structure and Key Identification

To support multiple devices, the DID structure remains consistent. The `controller` of a key is always the full DID of the entity (Employee, Individual). The key itself is identified by a fragment.

**Example `verificationMethod` for a device key:**

```json
{
  "id": "did:web:api.acme.org:employee:admin1@acme.org:role:isco-08|1120#macbook-pro-sig-key-id",
  "type": "JsonWebKey",
  "controller": "did:web:api.acme.org:employee:admin1@acme.org:role:isco-08|1120",
  "publicKeyJwk": {
    "kty": "AKP",
    "alg": "ML-DSA-44",
    "kid": "macbook-pro-sig-key-id",
    "pub": "..."
  }
}
```

## 4. Multi-Device Encryption Strategy

When sending an encrypted message (e.g., a response from the worker) to an entity with multiple registered devices, the system must ensure the entity can decrypt it on any of their devices.

- **Strategy**: **Multi-Recipient JWE**.
- **Flow**:
    1. The system retrieves the `didDocument` for the recipient.
    2. It finds all active `verificationMethod` entries intended for `keyAgreement` (encryption).
    3. It constructs a single JWE that is encrypted for all of these public keys. The content is encrypted once with a symmetric Content Encryption Key (CEK), and the CEK is then encrypted multiple times, once for each recipient key.
    4. The entity can then use the private key of any of their active devices to decrypt the message.

## 5. Key Rotation and Revocation

- **Rotation**: Key rotation will use a dedicated action, similar to DCR. The request must be signed by the key being deprecated (`old key`) and must contain the `jwk` of the `new key`. The system verifies the signature and replaces the old `verificationMethod` with the new one.
- **Revocation**: An entity admin or the user themselves can request to revoke a device key (e.g., if a device is lost). This request must be signed by another active key from a different device, proving the user's control. The system then removes the corresponding `verificationMethod` from the `didDocument`.

# Architecture: Legacy Flow and Managed Keys

This document describes the "Legacy" communication flow, which is designed for clients that do not have native cryptographic capabilities for DIDComm JWE/JWS messaging. It relies on a traditional HTTPS-based security model and is intrinsically linked to the concept of `Managed Keys`.

## 1. Core Concepts

- **Legacy Flow**: A communication pattern where the client sends and receives plain JSON/FHIR+JSON payloads over a mutually authenticated TLS (HTTPS) channel. Authentication is established via Bearer Tokens, not cryptographic signatures on the payload.
- **Managed Key**: A cryptographic key pair generated by the server on behalf of a user. The user does not have direct access to the private key material on their device. Instead, they authorize the server to use it on their behalf for specific operations.
- **The Connection**: An entity that only possesses a `Managed Key` **must** interact with the system via the `Legacy Flow` to access data encrypted for them. They cannot decrypt DIDComm messages on their own.

## 2. Detailed User Flow

This flow outlines how a user with a `Managed Key` accesses data that was encrypted for them.

### Step 1: Session Authentication

1.  **Action**: The user authenticates with the client application using a standard identity provider (e.g., Sign in with Google, Apple ID, eIDAS).
2.  **Result**: The client application receives an `id_token` (JWT) from the provider.
3.  **API Call**: The client makes all subsequent API calls with this token in the authorization header: `Authorization: Bearer <id_token>`.
4.  **Server Action**: The server's API gateway validates this token on every request to authenticate the user's session and confirm their identity (e.g., `alice@gmail.com`).

### Step 2: Cryptographic Operation Authorization

When the user needs to perform an action that requires their `Managed Key` (e.g., read an encrypted message, sign a document), the session token is not enough. A higher level of authorization is required.

1.  **Client Action**: The application prompts the user for a second factor. This consists of:
    a.  **Key Secret**: The user's portion of the `seed` for their `Managed Key`, often entered as a PIN or password.
    b.  **OTP**: A One-Time Password sent to the user's trusted device (phone/email).
2.  **API Call**: The client sends a request to a secure endpoint. This request includes the `Bearer id_token`, the user's `seed` fragment, and the OTP.
3.  **Server Action (KMS)**:
    a.  The server verifies the session (`id_token`) and the OTP.
    b.  If both are valid, it retrieves its own stored fragment of the `seed` for that user.
    c.  It combines the two fragments to **temporarily reconstruct the user's private `Managed Key`** in memory.
    d.  It performs the requested cryptographic operation (e.g., decrypts a `ConfidentialStorageDoc`).
    e.  **Crucially, it immediately discards the reconstructed private key** after the operation is complete.

### Step 3: Data Transmission

1.  **Server Action**: The server takes the plaintext data resulting from the cryptographic operation.
2.  **Response**: It sends the data back to the client as a standard `application/json` response body over the secure HTTPS channel. The response is **not** a JWE.

This flow provides a high level of security by ensuring the server never permanently stores the full private key, while still allowing users without crypto-wallets to interact with the secure data ecosystem.

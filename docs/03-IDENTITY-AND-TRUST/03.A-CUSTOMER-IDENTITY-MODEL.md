# Customer Identity Data Model

> **Related Documents:**
> *   **[Identity Bootstrapping and Trust Model (`IDENTITY_BOOTSTRAP_GUIDE.md`)](IDENTITY_BOOTSTRAP_GUIDE.md)**: Defines the high-level principles for all entities.
> *   **[FHIR Parameters Resource](https://build.fhir.org/parameters.html)**: The standard on which the `parameters` and `indexed` arrays are based.

## 1. Core Architecture: Unified and Encrypted Configuration

The customer identity model is designed for a high degree of security, flexibility, and privacy-preserving searchability. The entire configuration for a single customer is stored within a **single `ConfidentialStorageDoc`**, which is fully encrypted at rest in the tenant's vault.

The core of the model is the `CustomerConfig` object, which resides in the `content` field of the `ConfidentialStorageDoc` before encryption. This object logically separates identity data into three distinct containers, each with a specific purpose.

---

## 2. The Three Data Containers within `CustomerConfig`

Once the `ConfidentialStorageDoc` is decrypted by an authorized service, the `CustomerConfig` object reveals its three core components:

### 2.1. `indexed`: For Verifiable, Zero-Knowledge Search

*   **Type**: `ParamAttribute[]` (An array of `{ name, value, unique }` objects).
*   **Purpose**: To enable **secure, exact-match searching and verification** of identity attributes without revealing any information about the original data, including the types of attributes being stored.
*   **CRITICAL: Content is Fully Protected**: Within this array, **both the `name` and the `value` of each attribute are protected**. They are **not plaintext**. They are irreversible, keyed hashes (**HMACs**) of the original data. This prevents data leakage, as an attacker with database access cannot determine what types of identifiers a user possesses (e.g., they cannot see a plaintext "PPES" key and infer the user has a passport).
*   **Search Mechanism**: A search operation must first compute the HMACs of the attribute name and value it is looking for, and then query the database for an exact match of that protected pair.

### 2.2. `parameters`: The Source of Truth for Readable Data

*   **Type**: `ParamAttribute[]` (An array of `{ name, value, unique }` objects).
*   **Purpose**: To store the **original, human-readable plaintext values** of all attributes. This is the **only** place from which the system can retrieve and display a user's email address, phone number, or official identifier after a successful search.
*   **Relationship to `indexed`**: For every attribute that is searchable, there will be two corresponding entries:
    1.  An entry in `indexed` with the protected (HMAC) name and value for searching.
    2.  An entry in `parameters` with the original plaintext name and value for retrieval.

### 2.3. `claims`: For Primary, Single-Value PII

*   **Type**: `object`.
*   **Purpose**: To store primary, single-value, and sensitive Personal Identifiable Information (PII) that defines the core of the individual's identity (e.g., `birthDate`, `gender`).

## 3. Cryptographic Flow: How Data is Protected

The security of the `indexed` array relies on a robust Keyed-Hash (HMAC-SHA256) strategy that ensures data isolation between tenants.

### 3.1. HMAC Key Management

*   **HMAC Key per Entity**: Each entity with a security context (like the Host or a Tenant) has its own unique, randomly generated 32-byte HMAC key. This key is provisioned by the `KmsService` and stored alongside the entity's other keys (DEK, signing keys).
*   **`IKmsService` Responsibility**: The `KmsService` encapsulates all key management. Business logic (like a `Manager`) never handles raw keys. It requests an HMAC calculation by providing the plaintext data and the `entityVaultId` of the relevant tenant.

### 3.2. The Complete Flow

When the `CustomerManager` receives a `_batch` request, it performs the following for each identifiable attribute (e.g., a Spanish DNI represented by a `ParamAttribute` with a `name` property of `"NNES"` and a `value` property of `"12345678X"`) within the security context of a `tenantVaultId`:

1.  **Assemble `parameters` Array**: The original, plaintext attribute is added to the `parameters` array.
    ```typescript
    parameters.push({ name: 'NNES', value: '12345678X', unique: true });
    ```

2.  **Assemble `indexed` Array via `IKmsService`**:
    *   The manager calls the KMS to protect the **attribute's `name` property**:
        `protectedName = await kmsService.getHmacBase64Url('NNES', tenantVaultId);`
    *   The manager calls the KMS to protect the **attribute's `value` property**:
        `protectedValue = await kmsService.getHmacBase64Url('12345678X', tenantVaultId);`
    *   A new attribute object containing the protected `name` and `value` properties is added to the `indexed` array. The resulting strings will be in Base64UrlSafe format.
    ```typescript
    indexed.push({ name: protectedName, value: protectedValue, unique: true });
    ```

3.  **Assemble `CustomerConfig`**: The `claims`, `parameters`, and `indexed` arrays are combined into a single `CustomerConfig` object.

4.  **Encrypt and Store**:
    *   This `CustomerConfig` object is placed in the `content` field of a `ConfidentialStorageDoc`.
    *   The entire `ConfidentialStorageDoc` is then passed to the KMS, which encrypts the `content` into a JWE.
    *   The final, encrypted document is stored in the vault.

This unified model provides a robust and elegant solution, combining secure, privacy-preserving search with the functional necessity of data retrieval, all within a single, encrypted container.

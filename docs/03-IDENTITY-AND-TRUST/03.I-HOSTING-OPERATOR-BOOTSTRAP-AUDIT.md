# Hosting Operator Bootstrap Audit

Status: Audit Reference

## Purpose

This document is the audit-oriented reference for the current and target
bootstrap lifecycle of the GW Hosting Operator.

It separates:

- human/legal onboarding performed against ICA
- technical bootstrap performed by GW as the hosting operator runtime
- hosted-tenant publication and ledger-writing responsibilities

It is intentionally narrower than the general identity and onboarding guides.

## Scope

This document covers only the **hosting operator** bootstrap lifecycle.

It does not define:

- hosted-tenant dataset discovery inside `DigitalTwinProvider`
- per-tenant Fabric certificate issuance
- controller key custody inside GW

## Control Boundary

### Human controller boundary

The human controller of the hosting operator acts outside GW:

1. signs the onboarding PDF with its own means
2. authenticates against ICA, for example with Google login and `id_token`
3. sends the signed PDF and controller public key material to ICA
4. receives ICA-issued credentials/VCs for the hosting operator/controller flow

GW does **not** use its own KMS to sign on behalf of the human controller.

### GW runtime boundary

Once the hosting operator has the right legal/business activation context, GW
performs the technical runtime bootstrap required to operate as a host:

- technical authentication/bootstrap
- device/application registration
- CSR enrollment for the hosting operator certificate
- encrypted persistence of resulting host artifacts
- later hosted-tenant operation and ledger registration

## Canonical Separation

### A. Human/legal trust in ICA

Authority:

- ICA

Artifacts:

- legal/controller verification outcome
- operator-related VCs

Transport:

- external user journey, outside GW runtime signing

### B. Hosting operator technical bootstrap in GW

Authority:

- GW runtime acting as hosting operator

Artifacts:

- host technical session/bootstrap context
- host DCR result
- host CSR result
- host X.509 chain

Persistence:

- host vault storage, encrypted

### C. Hosted tenant operation

Authority:

- GW hosting operator

Artifacts:

- hosted tenant runtime config
- hosted tenant autodiscovery publication
- ledger writes performed by the hosting operator on behalf of hosted tenants

## Network Type vs Business Sector

The current model must distinguish two different dimensions:

### Network type

This applies to the **hosting operator / hosting service** technical operating
context.

Examples:

- `fabric`
- `pontusx`
- `dual`

This dimension determines:

- which technical bootstrap is possible
- which adapter is used
- which cryptographic/runtime artifacts are required
- whether ICA can issue a direct technical credential for that network type

### Business sector

This applies to **hosted tenants and published services**.

Examples:

- `health-care`
- `animal-care`
- `research-study`

This dimension determines:

- hosted-tenant routing
- service publication/discovery
- tenant/provider capability semantics

Audit rule:

- do not model hosting-operator network support as if it were a business sector
- do not model tenant business sector as if it were the host network type

## Current Architectural Rule

At the current phase:

- hosted tenants obtain VCs through ICA-backed activation/verification flows
- hosted tenants do **not** receive their own Fabric certificate by default
- the hosting operator is the current ledger writer for hosted tenants
- CSR/X.509 enrollment is currently a **host-only** flow

Future per-tenant certificate issuance may be added later, but it is not part
of the current baseline.

## ICA Support By Network Type

ICA technical enablement is network-type dependent.

### Fabric

If ICA supports `fabric` for the hosting service, ICA may issue the technical
enablement/credential path needed for the hosting operator to bootstrap and
operate against Fabric.

This is the direct technical-enablement case.

### Pontus-X

At the current phase, `pontusx` does not follow the same direct ICA CSR model
as Fabric.

Instead, Pontus-X operation is currently mediated through the
`globaldatacare.es` backend with an API key.

This is the mediated-backend case.

### Consequence

ICA does not need to expose one universal blockchain credential format for all
network types.

Instead:

- direct technical enablement may exist for `fabric`
- mediated API-key operation may exist for `pontusx`
- future network types may define their own technical operating contract

## Resolution Order For Host Certificate Material

The hosting operator must resolve its X.509 material in this order:

1. environment configuration
2. host vault storage
3. ICA technical bootstrap

### 1. Environment configuration

If host certificate material is already present in environment configuration,
GW may use it as bootstrap input.

Current known compatibility inputs include:

- `LEGACY_X509_DER_BASE64`
- `LEGACY_X509_CHAIN_BASE64`

This path is compatible but should not be the long-term primary production
source once vault persistence is working end to end.

### 2. Host vault storage

If the environment does not provide host certificate material, GW must attempt
to load it from the host vault.

The host vault is the canonical runtime persistence target after a successful
technical bootstrap.

### 3. ICA technical bootstrap

If the material is neither in environment configuration nor in the host vault,
GW must bootstrap the hosting operator against ICA.

That bootstrap is the technical sequence:

1. `_exchange`
2. `_dcr`
3. `_csr`

The output must then be persisted back into the host vault.

This certificate resolution order applies to direct certificate-based network
operation such as `fabric`.

For mediated-backend modes such as current `pontusx`, the equivalent runtime
resolution target is the technical API credential and adapter configuration,
not a Fabric-style host X.509 chain.

## Preconditions For ICA Technical Bootstrap

Before GW can run the host technical bootstrap, the following must already be
true:

1. the hosting operator human/controller journey has been completed against ICA
2. the hosting operator legal/business activation context already exists
3. the hosting operator is entitled/licensed to perform technical bootstrap
4. GW has the configuration needed to initiate host technical runtime actions

This means the following controller-owned steps are **not** performed by GW:

- signing the PDF
- proving controller identity with the controller private key
- authenticating the human controller as if GW were the controller

## Canonical Host Technical Sequence

### Step 1. Resolve bootstrap inputs

GW resolves whether host certificate material already exists:

- from env
- from host vault

If yes, no ICA bootstrap is needed.

### Step 2. Run `_exchange`

GW uses the hosting-operator runtime entitlement/context to obtain the
technical exchange result needed for the next bootstrap stages.

### Step 3. Run `_dcr`

GW registers the hosting operator technical client/device identity.

This is a technical runtime registration step. It is not the controller's human
signing proof.

### Step 4. Generate keypair and CSR

GW generates the host keypair locally and creates the CSR.

The private key remains inside the hosting environment and must not be uploaded
to ICA.

### Step 5. Run `_csr`

GW submits the hosting operator CSR to ICA.

Expected result:

- issued host certificate
- chain
- CA metadata if returned

This step is required only for network types that use direct CSR/certificate
bootstrap, such as `fabric`.

For mediated-backend modes such as current `pontusx`, the equivalent step is:

- resolve technical API credential
- resolve adapter endpoint and request context
- verify the backend is authorized to publish on behalf of the hosting service
  or hosted tenant

### Step 6. Persist host artifacts

GW stores the resulting host bootstrap artifacts in encrypted host vault
storage.

This includes the certificate chain and any associated technical bootstrap
material required for runtime publication.

### Step 7. Publish host well-known artifacts

After resolution/persistence, GW publishes the host public material via:

- `/.well-known/jwks.json`
- `/.well-known/x509.der`

### Step 8. Start hosted-tenant operations

Once the hosting operator bootstrap is complete, GW may:

- host tenants
- register hosted-tenant ledger actions
- publish provider-capable hosted tenants in host autodiscovery

## Hosted Tenant Flow After Host Bootstrap

Once the hosting operator is operational, the hosted-tenant flow is separate:

1. hosted tenant activation/verification
2. hosted tenant VCs from ICA-backed trust flow
3. host persistence of tenant runtime state
4. host publication of provider-capable tenants

The hosted-tenant flow must not be conflated with host CSR issuance.

## Network Adapter Modes

### Fabric-only mode

- host network type includes `fabric`
- ICA may issue direct technical enablement for Fabric
- host may use `_exchange -> _dcr -> _csr`
- host writes ledger state through the Fabric adapter

### Pontus-X-only mode

- host network type includes `pontusx`
- operation is currently mediated via `globaldatacare.es`
- authentication is currently based on an API key
- host publishes to Pontus-X through the mediated adapter/backend

### Dual mode

- host network type includes both `fabric` and `pontusx`
- the host may operate through both adapters
- direct Fabric bootstrap and mediated Pontus-X publication may coexist

## Pontus-X Mediated Backend Model

At the current phase, Pontus-X is not treated as a direct host-certificate
bootstrap equivalent to Fabric.

Instead, GW operates through an intermediary backend API exposed by
`globaldatacare.es`.

Expected properties:

- authenticated by API key or equivalent technical credential
- adapter-specific request/response contract
- publication may happen on behalf of:
  - the hosting service
  - a hosted tenant

This means the mediated backend must preserve request context clearly enough to
distinguish:

- who is the operating host
- which tenant, if any, is the publication subject
- which network adapter executed the publication

## Current Implementation State

### Confirmed current behavior

- GW bootstraps or loads the `host` tenant at startup
- host DID/JWKS/public artifacts are persisted and published
- host legacy X.509 compatibility config is still supported
- hosted tenants no longer trigger ICA CSR enrollment by default
- host autodiscovery publishes provider-capable tenants only

### Confirmed current rule

- `requestIcaEnrollment(...)` is now a host-only architectural concept
- hosted-tenant onboarding is no longer treated as a tenant certificate journey
- the intended model now distinguishes direct Fabric enablement from mediated
  Pontus-X backend publication

### Remaining unification gap

The startup contract still needs to be made explicit in code and docs as:

- use env host cert if present
- else use host vault cert if present
- else run host `_exchange -> _dcr -> _csr`
- then persist to host vault

This document defines that as the target canonical order.

## Storage And Key Protection Expectations

### Production expectation

Host certificate artifacts obtained from ICA must not depend permanently on
plaintext environment configuration.

Production expectation:

- private runtime keys remain under the hosting operator security boundary
- certificate artifacts are persisted in host vault storage
- confidential vault storage remains encrypted according to the configured
  protection model

### Compatibility expectation

Environment-provided certificate material remains valid as a bootstrap or
migration fallback, but not as the preferred long-term operational source.

## Ledger Responsibility

At the current phase:

- the hosting operator writes ledger records for hosted tenants
- hosted tenants do not each own a separate Fabric certificate lifecycle
- per-tenant certificate ownership is deferred to a later phase if needed

In mediated Pontus-X mode, publication may be executed by the backend on behalf
of the hosting operator or hosted tenant, but the hosting operator remains the
runtime owner of that publication request context.

## Discovery Responsibility

Discovery is split in two layers:

### ICA discovery

ICA is the trust/discovery source for trusted hosting operators.

### Host discovery

Each hosting operator publishes its hosted provider-capable tenants through:

- `GET /.well-known/dcat3/catalog`
- `POST /api/dataspace-discovery/providers`

Therefore:

- ICA identifies trusted hosting operators
- each host publishes the services offered by its hosted tenants

Network type support belongs to the hosting operator trust/technical profile,
not to the tenant business-sector publication layer.

## Audit Statements

The following statements should hold for the current intended model:

1. GW does not sign as the human controller.
2. Human/controller onboarding is external to GW runtime signing.
3. Host CSR bootstrap is a hosting-operator technical flow.
4. Hosted tenants do not currently receive their own Fabric certificate by
   default.
5. Hosted tenants receive trust/VC artifacts through ICA-backed activation or
   verification flows.
6. The hosting operator is the current ledger writer for hosted tenants.
7. Public hosted-tenant discovery is derived from host publication, not from
   per-tenant Fabric certificate ownership.
8. Hosting-operator technical support must be modeled by network type, not by
   tenant business sector.
9. Pontus-X is currently a mediated backend mode, not the same direct
   certificate bootstrap path as Fabric.

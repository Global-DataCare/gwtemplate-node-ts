# DID and URN Identifiers for Data Verification

This document outlines the structure of Decentralized Identifiers (DIDs) and Uniform Resource Names (URNs) used within the system, particularly for the end-to-end testing flow. The primary use case is to verify data hashes on a blockchain network, linking them to a `did:web`.

_Note: For improved readability in this document, DIDs and URNs are displayed with line breaks. In actual implementation, they are single, continuous strings without spaces or breaks._


## Professional DID Address

A Professional's DID is constructed using their email and role, based on the International Standard Classification of Occupations (ISCO-08).

-   **Structure:**
    ```
    did:web:api.acme.org:
    employee:email:<email>:
    role:ISCO-08:<role-code>
    ```
-   **Example:**
    ```
    did:web:api.acme.org:
    employee:email:receptionist1@api.acme.org:
    role:ISCO-08:4226
    ```
-   **Resolves to:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>`
-   **Well-Known URI:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>/.well-known/`

This DID resolves to a public, online endpoint. A professional may have multiple roles, each associated with different cryptographic keys.

## Individual DID Address

An Individual's DID is based on a UUID or a multibase-encoded value for brevity.

-   **Structure:**
    ```
    did:web:api.acme.org:
    individual:<uuid|multibase>:<value>
    ```
-   **Resolves to:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>`
-   **Well-Known URI:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>/.well-known/`

This DID resolves to a public, online endpoint. The identifier can be changed by the individual at any time.

## Emergency URN Identifier Use Case

In addition to their anonymous `did:web` identifier, a person can also use their phone number for emergency situations. The process mirrors the other examples: the URN is used locally to calculate a hash, which is then encoded (multiformat/multibase) and stored on the blockchain. Like other personal URNs, it does not resolve online.

This allows an individual to register both their DID and the hash of their phone number in their device's emergency settings (e.g., the "SOS Screen"). In an emergency, a first responder can see these identifiers and use them to look up the person's data index from a provider authorized by that individual.

-   **Example URN for Emergencies:**
    ```
    urn:<network>:global:
    mobile:E164:+34600123456
    ```

## URNs for Private Data Hashing


### URN with Private Identifier

-   **Purpose:** To generate a hash from a national or private identifier (e.g., DNI in Spain).
-   **Structure:**
    ```
    urn:<network>:global:
    identifier:NNES:<DNI>
    ```

### URN with Name

-   **Purpose:** To generate a hash from just the individual's name (following the ICAO 9303 standard).
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME
    ```

### URN with Name and Date of Birth

-   **Purpose:** To generate a hash from the individual's name (following the ICAO 9303 standard for machine-readable travel documents) and optional date of birth.
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME:
    dob:ISO:1980-12-30
    ```

### URN with Mobile Phone Number

-   **Purpose:** To generate a hash from a mobile phone number. The E.164 standard is recommended for international formatting.
-   **Structure:**
    ```
    urn:<network>:global:
    mobile:E164:+34600123456
    ```

### URN for Relationships (Related Person)

-   **Purpose:** To generate a hash representing a relationship between two individuals. It includes the primary individual's name and identifier, plus the related person's identifier and the relationship type.
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME:
    identifier:<id-type>:<id-value>:
    relatedperson:<ICAO9303|uuid|multibase|email|phone>:<value>:
    relationship:<code-type>:<code-value>
    ```

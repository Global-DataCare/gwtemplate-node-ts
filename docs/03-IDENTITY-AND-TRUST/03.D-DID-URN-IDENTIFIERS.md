 # DID and URN Identifiers for Data Verification

This document outlines the structure of Decentralized Identifiers (DIDs) and Uniform Resource Names (URNs) used within the system, particularly for the end-to-end testing flow. The primary use case is to verify data hashes on a blockchain network, linking them to a `did:web`.

## Public DID Resolution vs. Operational Endpoints

When a tenant publishes a DID Document:

- `didDocument.id` is the public `did:web` used for resolution.
- `alsoKnownAs` contains aliases such as hosted/operator forms, branded URLs, or URNs.
- `didDocument.service[].serviceEndpoint` contains the real invocable URL.

Client rule:

1. Resolve the public DID.
2. Read `service[].serviceEndpoint`.
3. Call that URL directly.

Do not reconstruct API URLs by concatenating `did:web` path segments.

_Note: For improved readability in this document, DIDs and URNs are displayed with line breaks. In actual implementation, they are single, continuous strings without spaces or breaks._


## Professional DID Address

A Professional's DID is constructed using their email and role, based on the International Standard Classification of Occupations (ISCO-08).

-   **Structure:**
    ```
    did:web:api.acme.org:
    employee:email:<email>:
    role:ISCO-08|<role-code>
    ```
-   **Example:**
    ```
    did:web:api.acme.org:
    employee:email:receptionist1@api.acme.org:
    role:ISCO-08|4226
    ```
-   **Resolves to:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>`
-   **Well-Known URI:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>/.well-known/`

This DID resolves to a public, online endpoint. A professional may have multiple roles, each associated with different cryptographic keys.

## Individual DID Address

An Individual's DID is based on a UUID or a multibase-encoded value for brevity.

-   **Structure:**
    ```
    did:web:api.acme.org:
    individual:<uuid|multibase>:<value>
    ```
-   **Resolves to:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>`
-   **Well-Known URI:**
    `https://api.acme.org/individual/<uuid|multibase>/<value>/.well-known/`

This DID resolves to a public, online endpoint. The identifier can be changed by the individual at any time.

## Emergency URN Identifier Use Case

In addition to their anonymous `did:web` identifier, a person can also use their phone number for emergency situations. The process mirrors the other examples: the URN is used locally to calculate a hash, which is then encoded (multiformat/multibase) and stored on the blockchain. Like other personal URNs, it does not resolve online.

This allows an individual to register both their DID and the hash of their phone number in their device's emergency settings (e.g., the "SOS Screen"). In an emergency, a first responder can see these identifiers and use them to look up the person's data index from a provider authorized by that individual.

-   **Example URN for Emergencies:**
    ```
    urn:<network>:global:
    mobile:E164:+34600123456
    ```

## URNs for Private Data Hashing


### URN with Private Identifier

-   **Purpose:** To generate a hash from a national or private identifier (e.g., DNI in Spain).
-   **Structure:**
    ```
    urn:<network>:global:
    identifier:NNES:<DNI>
    ```

### URN with Name

-   **Purpose:** To generate a hash from just the individual's name (following the ICAO 9303 standard).
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME
    ```

### URN with Name and Date of Birth

-   **Purpose:** To generate a hash from the individual's name (following the ICAO 9303 standard for machine-readable travel documents) and optional date of birth.
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME:
    dob:ISO:1980-12-30
    ```

### URN with Mobile Phone Number

-   **Purpose:** To generate a hash from a mobile phone number. The E.164 standard is recommended for international formatting.
-   **Structure:**
    ```
    urn:<network>:global:
    mobile:E164:+34600123456
    ```

### URN for Relationships (Related Person)

-   **Purpose:** To generate a hash representing a relationship between two individuals. It includes the primary individual's name and identifier, plus the related person's identifier and the relationship type.
-   **Structure:**
    ```
    urn:<network>:global:
    name:ICAO9303:SURNAME1<SURNAME2<<GIVEN<NAME:
    identifier:<id-type>:<id-value>:
    relatedperson:<ICAO9303|uuid|multibase|email|phone>:<value>:
    relationship:<code-type>:<code-value>
    ```

## DID Document Service `id` Formats

A DID Document's `service` array contains two distinct types of service definitions, each with a unique `id` format tailored to its purpose: Standard Discovery services and Internal Business services. This distinction is critical to understanding the system's architecture.

### 1. Standard Discovery Service IDs

-   **Purpose:** To allow external clients and standard W3C-compliant tools to discover an entity's core endpoints, such as its public keys (`jwks.json`) or the DID document itself.
-   **Format:** The `id` is constructed by appending a fragment (`#`) to the entity's full `did:web` identifier.
-   **Structure:** `{did}#{service-name}`
-   **Key Principle:** The `did:web` specification dictates that colons (`:`) in the DID path map directly to slashes (`/`) in the corresponding HTTPS URL.
-   **Examples:**
    -   **Host DID:**
        -   DID: `did:web:gateway.example.com`
        -   Service ID: `did:web:gateway.example.com#jwks`
        -   This corresponds to the URL path: `https://gateway.example.com/`
    -   **Hosted Tenant DID:**
        -   DID: `did:web:gateway.example.com:acme:cds-es:v1:health-care`
        -   Service ID: `did:web:gateway.example.com:acme:cds-es:v1:health-care#did-document`
        -   This corresponds to the URL path: `https://gateway.example.com/acme/cds-es/v1/health-care/`
-   **Rationale:** This format is defined by the W3C DID Core specification. It creates a globally unique and resolvable identifier for a service. The `populateDidDocumentServices` utility is responsible for creating these standard service IDs.

### Service Endpoint Publication Rule

- `#did-document` and `#jwks` are published on the public resolution base URL.
- Business/API services are published on the operational base URL.
- The operational base URL can differ from the public DID domain.

Example split:

- Public DID: `did:web:public.acme.org`
- Public DID document: `https://public.acme.org/.well-known/did.json`
- Operational SMART token endpoint: `https://operator.example.net/acme/cds-es/v1/health-care/identity/openid/smart/token`

### 2. Internal Business Service IDs

-   **Purpose:** To provide a mechanism for the gateway's internal API router to authorize requests. This ID is **not** for external discovery and is never used as a fragment identifier.
-   **Format:** The `id` is a programmatic string built from the components of the API request path, joined by colons.
-   **Structure:** `{version}:{sector}:{section}:{format}`
-   **Key Principle:** The `{format}` component **MUST** follow Reverse DNS notation (e.g., `org.schema`, `org.hl7.fhir.r4`).
-   **Examples:**
    -   `v1:health-care:entity:org.schema`
    -   `v1:test:registry:org.schema`
-   **Rationale:** When a request is made to an endpoint like `/acme/cds-es/v1/health-care/entity/org.schema/Employee/_batch`, the API router constructs the service ID from the URL parameters. It then checks if a service with this exact ID exists in the tenant's `didConfig`. If a match is not found, the router rejects the request with a `404 Not Found`, effectively enforcing that the tenant is authorized to use that specific API route. The `createDidServiceIdBase` utility is responsible for creating these programmatic IDs.

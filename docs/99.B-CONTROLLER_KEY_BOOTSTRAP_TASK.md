# Controller Key Bootstrap Task

Purpose: executable instructions for a separate implementation/review thread.

This task clarifies the separation between:

- device / portal / app transport identity
- human controller / professional / individual / related-person signing identity
- DCR registration responsibilities
- activation-time DID Document publication responsibilities

This file exists separately from:

- [99.A-DID_ENDPOINT_RESOLUTION_TASK.md](/Users/fernando/GITS/gdc-workspace/gwtemplate-node-ts/docs/99.A-DID_ENDPOINT_RESOLUTION_TASK.md)

because this is primarily an identity/bootstrap/security problem, not only an endpoint publication problem.

## Core Question

What exactly is DCR registering?

The implementation thread must answer this explicitly for each runtime form:

1. shared portal web
2. confidential native app
3. backend-controlled node client

## Working Model To Validate

Default working assumption unless existing code/tests prove otherwise:

### 1. Shared portal web

- DCR registers the portal technical client identity
- the portal uses one transport key set for DIDComm/JOSE communications
- the portal technical identity is shared across many human users
- the controller/professional/individual person identity is separate

Implication:

- DCR from a shared portal should not be treated as "the controller's personal signing key registration"
- it is primarily the portal transport/runtime registration

### 2. Confidential native app

- DCR may register the app/device technical identity for one user profile
- that runtime may be tightly coupled to the user profile
- but even then, transport keys and person-signing keys should still be separated by purpose

Implication:

- the same wallet/runtime may manage both
- but the model must still distinguish:
  - communication transport keys
  - human authorization/signing keys

### 3. Human/person signing identity

The person identity is for:

- controller
- professional
- individual
- related person

These keys are relevant for:

- proof-bearing actions
- authorization decisions
- SMART token subject binding
- consent/controller-controlled operations

These are not automatically the same keys as the transport keys used by the portal/app runtime.

## Main Design Decision To Close

For an organization activation flow:

- can CORE GW create and publish the tenant organization DID Document before it knows the controller's public signing keys?

Recommended answer:

- yes for the organization/provider transport DID Document
- no for the controller person DID Document if that controller identity is meant to be published/bound immediately

Meaning:

1. the tenant/provider DID Document can be created by GW using connector-managed transport keys
2. the controller binding is a separate concern and may require controller public JWK/JWKS or `kid`

The same question applies to an individual controller:

- the individual/provider-side DID Document can exist
- but a controller-person DID binding may require explicit user/controller keys

## Practical Recommendation

### Organization

At activation/bootstrap time, distinguish:

1. Organization/provider DID Document
- published by GW
- contains transport/service keys managed by connector/GW
- contains operational service endpoints

2. Controller person DID Document
- published only if the flow already has the controller public keys
- otherwise controller binding remains pending/incomplete

### Individual

At bootstrap time, distinguish:

1. Individual/provider-side record and provider-hosted DID/service context
- may be created as part of the hosted index flow

2. Individual controller person DID Document
- only if the controller's public signing identity is already available

## What DCR Should Mean

The implementation thread must decide whether current DCR semantics are:

1. technical client registration only
2. technical client + human controller registration together
3. two separate steps that are currently conflated

Recommended target:

- DCR = technical client registration
- controller/professional/individual signing identity binding = separate explicit step or explicit activation payload material

## What `_activate` Must Clarify

The implementation thread must inspect current `_activate` and adjacent flows and answer:

1. Does `_activate` already receive controller public key material?
2. If yes, where?
   - VP payload
   - VC credentialSubject
   - DIDComm `meta.jws.protected.jwk`
   - explicit request claim
   - DCR result reference
3. If not, what is the minimum additional input needed?

Candidate forms:

- controller public JWK
- controller JWKS
- controller signing `kid`
- controller DID
- controller DID Document reference

## Current Risk To Avoid

Do not accidentally do either of these:

1. publish organization/provider DID Document using controller personal keys as if they were tenant transport keys
2. assume that a portal DCR registration already proves the controller person identity

## What The SDK Must Be Able To Hold

After bootstrap, the SDK should be able to keep available:

### Shared portal web

- portal transport DID / DID Document / JWK set
- active actor DID(s) for human users
- resolved organization/provider DID Document

### Confidential native app

- app/device transport DID / DID Document / JWK set
- actor/person signing DID / DID Document / signing keys
- resolved organization/provider DID Document

## Questions The Thread Must Answer

1. Is DCR for device/app transport identity, person identity, or both?
2. Can organization DID Document publication happen before controller person-key binding?
3. Can individual-provider DID publication happen before controller person-key binding?
4. Which public keys must be present in `_activate` to finalize human/controller binding?
5. Which DID Documents are mandatory outputs after each phase?

## Acceptance Criteria

The task is done when:

1. The docs clearly distinguish technical transport identity from person signing identity.
2. DCR semantics are explicit.
3. `_activate` input requirements for controller key binding are explicit.
4. It is clear whether organization/individual DID Documents can be published before controller binding is complete.
5. SDK responsibilities are explicit for:
   - storing/generating transport keys
   - storing/importing person signing keys
   - resolving organization/individual DID Documents

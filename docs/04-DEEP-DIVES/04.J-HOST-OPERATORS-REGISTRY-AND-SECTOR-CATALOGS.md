# Node Operator Catalog and Tenant Sector Catalogs

## Status

Proposed

## Date

2026-05-28

## Problem

The gateway currently mixes or risks mixing three distinct publication concerns:

1. Host-level discovery of the node operator (`gwtemplate`) and the tenant
   organizations it hosts.
2. Tenant-level sector catalog publication.
3. Sector-specific metadata profiling such as `HealthDCAT-AP Release 7`.

These are not the same thing:

- a node operator catalog is not a tenant sector dataset catalog;
- a tenant sector dataset catalog is not the same artifact as the host broker
  catalog;
- `HealthDCAT-AP Release 7` applies to the shape of sector metadata, not to the
  node operator catalog itself.

At the same time, `gwtemplate-node-ts` already has:

- host-level discovery behavior;
- DCAT tests for `/dcat3/catalog/dcat.json`;
- tenant DID `CatalogService` publication;
- service capability projection from onboarding claims and persisted
  `Service.serviceType` values during `/_activate`.

Therefore the missing piece is not â€śput everything into ICA firstâ€ť, but rather
to formalize the host-side node operator catalog in `gwtemplate-node-ts` and
make host discovery and tenant catalog publication separate first-class flows.

## Decision

### A. Node operator catalog lives in GW

`gwtemplate-node-ts` is the runtime node operator. It is therefore the first
runtime owner of the host-side discovery catalog describing:

- the node operator itself;
- the tenant organizations hosted by that node operator that are eligible for
  discovery publication.

This is a runtime catalog and backing registry, not merely a static signed
file.

### B. Publication in the host catalog is derived from activation

Tenant organizations that appear in the host catalog do not require a separate
catalog-specific verification flow. Publication must be derived from the
existing onboarding lifecycle and, in particular, from the effective service
capabilities known at `/_activate`.

The relevant source of truth is the service metadata materialized at activation,
especially `Service.serviceType` or its persisted normalized equivalent.

`/_verify` and the preceding onboarding steps remain relevant for identity and
evidence validation, but host catalog publication should be a projection over
activated tenant organizations, not a parallel approval registry.

The key point is semantic: host catalog inclusion is a derived runtime
projection, not a second independent lifecycle.

Practical sequence:

1. GW completes onboarding and activation for the tenant.
2. GW completes any technical client bootstrap such as `Token/_exchange` and
   `Device/_dcr` for the calling app/device.
3. If the tenant or host needs ICA-backed X.509 material, GW brokers the CSR
   enrollment with ICA.
4. Only then does GW publish the tenant in the host catalog if the tenant is
   operational and its `Service.serviceType` denotes a provider capability.

### C. Host discovery catalog is node-operator-level only

The host-level discovery catalog is the node-operator discovery surface.

Canonical endpoint:

- `/.well-known/dcat3/catalog`

Compatibility alias:

- `/dcat3/catalog/dcat.json`

This catalog is allowed to expose `dcat:Dataset`, `dcat:DataService`, or both,
according to the host discovery model. It does not represent a HealthDCAT-AP
tenant sector data catalog.

Only tenant organizations that actually offer discoverable services should be
included. Reader-only capabilities must be excluded from service
autodiscovery/publication.

### D. Tenant sector catalogs remain tenant-scoped

Each tenant sector catalog must remain tenant-scoped.

Current implemented DSP request endpoint:

- `POST /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/request`
- `GET /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/datasets/{id}`

This is the current GW CORE local binding for DSP catalog interactions. The DSP
standard HTTPS binding paths are `/catalog/request` and
`/catalog/datasets/{id}`.

Future target backlog, if a tenant read-only catalog resource is introduced:

- `/{tenantId}/cds-{jurisdiction}/v1/{sector}/catalog`

Any future `/catalog` resource must be documented as an additional artifact,
not as a silent replacement for the current tenant DID `CatalogService`
contract.

### E. Sector profile rules

For tenant sector catalogs, the operational baseline is `DCAT3`.

`HealthDCAT-AP Release 7` may be applied as an additional sector profile when a
deployment needs health-oriented dataset semantics or a compatible local
interoperability profile. That profile choice must remain optional and
deployment-specific, including for `animal-*` and `onehealth-*`.

The safe cross-repository rule is:

- `DCAT3` is the required operational baseline.
- `HealthDCAT-AP Release 7` is optional profile guidance, not the baseline host
  or ICA contract.

## Publication Rule

Host catalog publication must be computed from activated tenant organizations.

Initial inclusion rule:

- include tenant organizations with service capabilities that denote an exposed
  provider/service role;
- exclude reader-only capabilities from autodiscovery.

At the time of writing, this means:

- include:
  - `IndexProvider`
  - `DigitalTwinProvider`
- exclude:
  - `IndexReader`
  - `DigitalTwinReader`

`DigitalTwinIndex` must be evaluated according to its real semantics:

- include it if it denotes a discoverable service exposed to third parties;
- exclude it if it denotes only internal indexing capability.

The durable rule is therefore:

- publish only tenant organizations whose `Service.serviceType` denotes an
  effective externally discoverable service offering.

## Data Model

The host catalog may be backed either by direct runtime projection or by a
materialized cache/projection record similar to:

```ts
type HostedTenantCatalogRecord = {
  id: string;
  tenantId: string;
  tenantDid: string;
  legalName: string;
  taxId: string;
  jurisdiction: string;
  serviceBaseUrl: string;
  sectorsPublished: string[];
  serviceTypeTokens: string[];
  projectionSource: 'activate-service-capabilities-v1';
  activationRecordId: string;
  status: 'published' | 'revoked' | 'suspended';
  publishedAt: string;
  updatedAt: string;
};
```

The exact storage backend may follow the same adapter style already used in GW
collections and caches.

## Endpoint Plan

### 1. Host node operator discovery catalog

- `GET /.well-known/dcat3/catalog`
- `GET /dcat3/catalog/dcat.json` as compatibility alias

### 2. Tenant sector catalog

- `POST /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/request`
- `GET /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/datasets/{id}`

Optional additional artifacts or aliases:

- `GET /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/dcat.json`
- future `GET /{tenantId}/cds-{jurisdiction}/v1/{sector}/catalog` if introduced
  as a read-only catalog artifact

## DID Service Publication

For tenants, DID `CatalogService` publication remains tenant-scoped.

For the host, the DID document may publish a host-level `CatalogService`
pointing to:

- `/.well-known/dcat3/catalog`

This service advertises node operator discovery and discovery of hosted tenant
organizations, not tenant sector dataset ownership.

The DID document remains the primary machine-readable publication surface for
service discovery. A future `/.well-known/api-catalog` may complement this for
HTTP/API discovery, but it must not be treated as a replacement for DID service
publication.

## Why This Order

This work should start in `gwtemplate-node-ts` before ICA because:

- GW already owns the runtime HTTP discovery layer.
- GW already has tests for host DCAT discovery.
- GW already projects service capability tokens into discovery artifacts.
- GW can later sync, attest, or federate this catalog into ICA without blocking
  current implementation.

## Migration Guidance

Phase 1:

- implement host catalog projection from activated tenant service capabilities;
- keep legacy catalog route aliases;
- keep current tenant discovery alive.

Phase 2:

- add synchronization or attestation with ICA-managed trust registry if desired;
- optionally sign exported host catalog snapshots.
- optionally add `/.well-known/api-catalog` as a complementary HTTP discovery
  artifact.

## Summary

- node operator catalog: in GW first
- host catalog: `/.well-known/dcat3/catalog`
- tenant catalog request endpoint:
  `/{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/request`
- host publication source of truth: activated tenant `Service.serviceType`
- readers excluded from service autodiscovery
- `DCAT3` is the baseline; `HealthDCAT-AP Release 7` remains optional profile
  guidance for tenant sector catalogs

# Fabric Multicloud Blueprint (UNID Governance)

This document defines the multi-cloud deployment plan for the Hyperledger Fabric
network, aligned with UNID governance and the channel taxonomy agreed in this
project.

## Goals

- Multi-cloud: GKE (UNID), GKE (Connectech), Arsys (Accuro).
- Environment separation: `test` and `prod` in each cluster.
- Deterministic PKI chain: Root CA → ICA → Host/Member (seeded).
- UNID in all channels as governance.
- Expandable orderer set (prod: 1 → 5 orderers).

## Organizations (MSPs)

- UNID (governance, all channels)
- Accuro (EU, Arsys)
- Connectech (AMER/APAC, GKE)
- ConnectHealth‑EMEA (future MSP, not yet in v1)

## Namespaces (each cluster)

- `test-fabric-v3`, `prod-fabric-v3`
- `test-gdc-v1`, `prod-gdc-v1`

Fabric components live in `*-fabric-v3`. The connector/host service lives in
`*-gdc-v1`.

## Channel governance policies (v1)

- UNID is a member of all channels.
- `global-index-*`:
  - write: Connectech (owner) + Accuro (agreement) + UNID (override).
- `eu-identity-*`:
  - write: Accuro (EU host) + UNID.
- `na-health-*`:
  - write: Connectech + UNID.
- `eu/na-vet-*`:
  - write: hosts approved by UNID (Accuro EU, Connectech NA/APAC).
- `global-identity-*`:
  - write: UNID + authorized orgs (by chaincode policy).

Endorsement policies are enforced per channel and per chaincode. UNID retains
final approval of chaincode definitions.

## Test-network policy (current)

Only **test-network** is enabled right now. When an organization registers with
`section = test-network`, the host will:
- verify the required documents (e.g., terms acceptance),
- request certificate issuance from the **test ICA**,
- and bind the tenant to the **test Fabric namespace**.

Mapping:
- `section = test-network`
  - ICA: `test-eur-ica.unid.online`
  - Kubernetes namespace: `test-fabric-v3`

Production enrollment is **not enabled yet**; no certificates are issued for
`section = production` at this time.

## PKI / Chain of Trust

Scripts live under `scripts/` (see `README.md`):
- Root CA: `pki:root`
- ICA: `pki:ica`
- Host: `pki:host`
- Member: `pki:member`

Outputs are in `artifacts/<env>/...` and MUST NOT commit private material.
ICA outputs are stored under `artifacts/<env>/pki-ica/<ICA_MSP_ID>/` to allow multiple ICAs.

## Deployment layout

UNID (GKE‑ES):
- Root CA + ICA

- UNID peers
- gdc host

Connectech (GKE‑NA):
- ICA (optional if using UNID ICA)
- Connectech peers
- gdc host

Accuro (Arsys‑EU):
- ICA (optional if using UNID ICA)

## Bootstrap sequence (test/prod)

1) Generate PKI (Root CA, ICA, Host/Member) with `--env test|prod`.
2) Generate `configtx.yaml` from `fabric-multicloud/configtx/configtx.yaml`.
3) Run `fabric-multicloud/scripts/10-generate-genesis.sh`.
4) Run `fabric-multicloud/scripts/11-generate-channel-txs.sh`.
5) Bring up orderer + peers in each cluster.
6) For each channel:
   - `20-create-channel.sh <channel>`
   - `21-join-channel.sh <channel>` per org
   - `22-set-anchor-peers.sh <channel>` per org
7) Deploy chaincodes via `30-deploy-chaincode.sh`.

## GKE quickstart (UNID demo)

1) Copy config:
   - `cp private-deploy.config.example private-deploy.config`
2) Edit `private-deploy.config` with project/region/cluster and MSP paths.
3) Source it:
   - `source private-deploy.config`
4) Create the cluster:
   - `fabric-multicloud/scripts/01-gke-create-cluster.sh`
5) Create namespaces:
   - `fabric-multicloud/scripts/02-k8s-bootstrap.sh`
6) Create secrets from MSP/TLS artifacts:
   - `fabric-multicloud/scripts/03-k8s-create-secrets.sh`
7) Deploy Fabric pods:
   - `fabric-multicloud/scripts/04-k8s-deploy-fabric.sh`

Fabric CA:
- Root + ICA are deployed separately (`fabric-ca-root`, `fabric-ca-ica`).
- Admin credentials and CA data/config are stored in K8s secrets.
- Example configs are under `fabric-multicloud/ca-config-examples/`.

MSP/TLS enrollment:
- Deploy CA pods first.
- Port-forward ICA (example):
  - `kubectl -n test-fabric-v3 port-forward svc/fabric-ca-ica 8054:8054`
- Enroll orderer/peer MSP/TLS:
  - `CA_URL=localhost:8054 CA_TLS_CERT=artifacts/test/fabric-ca-server-ica/<ICA_MSP_ID>/ca-cert.pem`
  - `CA_ADMIN_USER=... CA_ADMIN_PASS=... ORDERER_HOST=orderer.unid.local PEER_HOST=peer0.unid.local OUT_DIR=artifacts/test/enroll/UNID`
  - `fabric-multicloud/scripts/06-enroll-msp.sh`

## Orderer scaling (prod migration)

Phase 1:
- Single UNID orderer (Raft) for initial launch.

Phase 2:
- Add 4 more orderers (multi-org).
- Update consenters in system channel configuration.
- Rotate TLS certs and update `EtcdRaft` section.
- Verify new orderers catch up before removing the old config.

## Onboarding new orgs

1) Generate host/member certs for the org.
2) Add org MSP to `configtx.yaml`.
3) Create channel config update for each channel the org joins.
4) Deliver MSP material + TLS certs to their cluster.
5) Join peers and set anchor peers.

## K8s templates

Templates live under `fabric-multicloud/k8s/` and are placeholders. You must:
- replace secret data with real MSP/TLS material
- set storage classes
- wire hostnames / DNS
- configure load balancers and firewall rules

## gdc host / connector

The connector runs in `*-gdc-v1` namespaces and must be configured to:
- connect to the local peer
- use per-tenant certificates
- enforce channel access by tenant policy

# 04.F Persistence Patterns

This document outlines the core architectural patterns for data persistence, focusing on the registration and management of tenants.

## Tenant Creation and Storage Flow

The system makes a clear distinction between a tenant's **logical identifier (`vaultId`)** and its **physical storage identifier (`collectionName`)**. Understanding this separation is critical for development and testing.

The correct flow for creating a new tenant is as follows:

1.  **Calculate Logical ID**: The business logic manager (e.g., `HostingManager`) calculates the tenant's logical `vaultId`. This is typically a combination of the tenant's sector and its unique `alternateName` (e.g., `health-care_acme`).

2.  **Check for Existence**: The manager calls `vaultRepository.vaultExists(vaultId)` to check if this logical tenant is already registered. The repository implementation is responsible for checking this efficiently (e.g., by checking for a registration document).

3.  **Calculate Physical ID**: If the tenant does not exist, the manager calculates the physical `collectionName`. This is a unique, often obfuscated, identifier derived from the tenant's verifiable claims (e.g., jurisdiction, tax ID). Example: `ES_TAX_B0011233_health-care`.

4.  **Create Physical Storage**: The manager calls `vaultRepository.createNewVault({ id: collectionName })` to provision the physical storage space (e.g., a top-level collection in Firestore).

5.  **Create Registration Document**: The manager creates the tenant's registration document (`tenantRegistrationDoc`). The `id` of this document **MUST** be the logical `vaultId`.

6.  **Persist Registration Document**: This registration document is then saved into the **host's** physical collection, inside a dedicated `tenants` section. The repository implementation ensures this document is stored in a way that makes the `vaultExists` check efficient.

7.  **Persist Tenant-Specific Data**: All other tenant-specific documents (e.g., employees, services, credentials) are saved directly into the tenant's own physical storage, identified by its `collectionName`.

This pattern ensures that the system can quickly verify the existence of a tenant without needing to know the details of its physical storage location, while keeping all of a tenant's own data neatly partitioned.

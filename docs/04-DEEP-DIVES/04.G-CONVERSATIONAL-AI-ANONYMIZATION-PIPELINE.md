# Conversational AI & Anonymization Pipeline (Design)

This document describes a proposed architecture for:

- Storing conversational messages as confidential `Communication`-style entries (claims-first).
- Deriving anonymized artifacts for research/AI usage.
- Producing observation bundles with `meta.tag[]` (ledger-safe metadata) while ensuring privacy-by-design.

It is written for:
- API integrators (what goes where, what to send/expect).
- Security auditors (threat model and controls).
- AI ethics reviewers (data minimization, governance, and human oversight).

## 1) Problem statement

We have a multi-tenant host with tenant organizations; tenants manage individuals (often through a â€śfamily organizationâ€ť model).

We need to:

1. Store raw conversations safely (PII is expected in free text).
2. Allow draft/resume behavior (conversation history is stateful).
3. Derive structured clinical and non-clinical signals (e.g., Observations) from conversations.
4. Share selected, non-PII metadata (e.g., codes, cohort info) for data space / research flows.
5. Guarantee that PII never lands on an immutable ledger even if upstream anonymization fails.

## 2) Core design principles

### 2.1 Stateless AI, stateful storage

LLMs are treated as *stateless* workers. All state lives in storage:
- Conversation history is stored (confidential).
- Subject-level policies are stored (confidential).
- Derived bundles and indexes are stored (confidential for tenant; optionally separate research store).

If the AI service restarts, nothing is lost: it recomputes context from storage (often using summaries and/or a small window, not full history).

### 2.2 Isolation scope: `tenantId + subjectDid`

All conversational processing is scoped by:
- `tenantId` (security boundary)
- `subjectDid` (individual boundary)

No AI request may mix content across different scopes.

### 2.3 Defense in depth for privacy

- At-rest encryption for all sensitive conversation content.
- Blind indexing (HMAC) for searchable keys (`subject`, `identifier`, etc.).
- Post-processing allowlists for anything that may be published/shared externally.
- Chaincode/ledger enforcement that strips/rejects any free-text fields.

## 3) Data model (high level)

### 3.1 ConfidentialStorageDoc structure

Storage uses `ConfidentialStorageDoc`:
- `content`: the full â€śdata entryâ€ť payload (e.g., `meta.claims`, optional `resource`, etc.). This is encrypted into `jwe`.
- `indexed.attributes`: selected keys for blind queries (HMAC protected).
- `meta`: non-sensitive doc metadata.

Note: `meta` is visible without decryption, so it must be treated as non-PII or â€śpolicy-approvedâ€ť.

We currently support:
- `audit` (traceability metadata, optional)
- `tag[]` (FHIR Coding-like tags)
- `research.jurisdiction`
- `research.yearOfBirth`
- `research.gender`
- `research.sexAtBirth`

### 3.2 Two stores (tenant vault vs research/AI store)

- **Tenant vault (confidential):** raw conversations and full derived content, encrypted at-rest.
- **Research/AI store (policy-dependent):** anonymized text and/or derived metadata for cross-org sharing.

Do not store raw text in a shared ledger.

## 4) Processing pipeline

### 4.1 Step A â€” Ingest & store a Communication

Client sends a DIDComm/FAPI-secured request containing a `data[]` entry with:
- `type: "Communication-..."` (or equivalent)
- `meta.claims` (claims-first payload)
- optional `resource` (FHIR-style object, if present)

Backend persists the data entry as a `ConfidentialStorageDoc`:
- Encrypt `content` â†’ `jwe`
- HMAC-protect `indexed.attributes`

### 4.2 Step B â€” Anonymize

An anonymization service produces:
- anonymized text (no direct identifiers, no phone numbers, etc.)
- audit record / policy version reference

Implementation detail: the anonymizer can be LLM-based (Ollama/Gemma, OpenAI, etc.) or deterministic.
The interface must remain stable and provider-agnostic.

### 4.3 Step C â€” Derive Observations (bundle)

From anonymized text we derive an Observation bundle:
- Observations are represented as a FHIR Bundle (`body.resourceType="Bundle"`) with `body.entry[]`.
- Derived research/routing tags are attached at the bundle level in `body.meta.tag[]` (not inside each resource), so clients can display/route without altering clinical resources.

#### 4.3.1 `meta.tag[]` schema (ledger-safe)

Each tag is a flattened â€śFHIR meta.tag-likeâ€ť object:

```json
{
  "id": "Observation[0].code",
  "system": "SNOMED",
  "code": "48694002",
  "version": "optional",
  "display": "optional free text",
  "userSelected": true
}
```

Rules:
- `id` points to the claim key the tag annotates and MUST be unique within the bundle; index by resource type when needed (e.g., `Observation[0].code`, `Observation[1].date-when`).
- `display` is allowed for UI/debugging but MUST NOT be written to any immutable ledger.

### 4.4 Step D â€” Publish to ledger (optional)

If publishing to Hyperledger Fabric (or other immutable log), the publishing layer MUST:
- allowlist only `id/system/code/version/userSelected`
- strip `display` (and any unknown fields)
- enforce maximum lengths and safe character sets

This provides a hard guarantee even if upstream anonymization fails.

## 5) Provider-agnostic services (backend)

Recommended backend interfaces:

- `ConversationStore` (vault repository): read/write Communication entries by subject scope.
- `SubjectPolicyStore`: rules/policies versioned per subject/family-admin.
- `AnonymizationService`: provider-agnostic (`Ollama`, `OpenAI`, etc.).
- `DerivationService`: converts anonymized text â†’ Observation bundle.
- `LedgerPublisher`: strips/validates and submits metadata-only payloads.

### 5.1 Local Ollama (development/demo)

The backend can call a local Ollama instance via HTTP.

Environment variables (see `env.example`):
- `LLM_PROVIDER=ollama`
- `OLLAMA_BASE_URL=http://127.0.0.1:11434` (or `http://ollama:11434` in docker-compose)
- `OLLAMA_MODEL=gemma2:latest` (or any model you have pulled)

## 6) Human oversight and governance

For high-risk pipelines (research sharing), include:
- manual review workflow (family admin / professional) for approving anonymized transcripts or derived bundles.
- audit trails (who approved, when, policy version).
- â€śright to be forgottenâ€ť procedures consistent with the vault/index model.

  - conversion rules and validation
- Backend unit tests that:
  - mock the LLM provider
  - verify allowlist/strip behavior before ledger write

## 9) Default policy for digital twins and host AI access

### 9.1 Ownership and default access

- **Digital twin artifacts are tenant-local by default.**
- **Host access to tenant twin content is denied by default** (`default deny`).
- Any host-side AI processing over tenant twin content requires explicit opt-in policy.

### 9.2 Recommended access levels

Use explicit policy levels instead of implicit "host can read":

1. `none` (default)
   - Host cannot read tenant twin content.
   - Host may only process non-sensitive operational metadata.
2. `aggregate`
   - Host can receive aggregate/anonymized features (no direct identifiers).
   - Intended for cross-tenant analytics and generic model quality metrics.
3. `pseudonymized`
   - Host can process pseudonymized records under strict contract and retention.
   - Requires explicit tenant opt-in and subject/legal basis where applicable.
4. `identified`
   - Host can process identified content only for narrowly scoped, auditable support workflows.
   - Not recommended as a default AI mode.

### 9.3 Processing model

- Preferred model: **compute-to-data** at tenant side (move model, not raw data).
- If proxy processing is required:
  - enforce minimum-necessary fields,
  - define purpose and retention window,
  - record user-visible audit events for each access/decrypt/transform.

### 9.4 Catalog implications (host broker vs tenant catalog)

- Tenant catalogs should publish data products/endpoints and policy metadata, not raw PHI.
- Host catalog acts as federated broker of tenant product metadata.
- Host broker discovery does not imply host data custody.

### 9.5 Policy contract fields (minimum)

For each AI-relevant product/scope, store:
- `policyId`, `version`, `ownerTenantId`
- `accessLevel` (`none|aggregate|pseudonymized|identified`)
- `purpose` (bounded use)
- `retention` (TTL / deletion SLA)
- `consentRef` or legal basis reference
- `auditEnabled=true` with endpoint scope for user-visible logs

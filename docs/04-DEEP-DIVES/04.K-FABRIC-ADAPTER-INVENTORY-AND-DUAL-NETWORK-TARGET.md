# Fabric Adapter Inventory and Dual-Network Target

Status: Audit and Implementation Inventory

## Purpose

This document records:

- the existing Hyperledger Fabric integration already present in GW CORE
- the reason Fabric must remain supported
- the target dual-network model in which `Fabric` and `Pontus-X` may coexist

It exists to avoid re-designing the Fabric adapter as if it did not already
exist.

## Why Fabric Must Remain Supported

The SEDIA memory and planning baseline states that Hyperledger Fabric is part
of the expected network foundation.

Therefore:

- Fabric support is not optional from the architectural baseline point of view
- GW CORE must continue to support direct Fabric operation
- any future Pontus-X support must be additive, not a silent replacement for
  Fabric

## Current Gap

Although Fabric support exists, it is not yet unified with the newer
hosting-operator bootstrap model.

The main missing link is not the existence of a Fabric adapter.

The missing link is the explicit integration between:

- hosting-operator technical bootstrap
- host certificate resolution order
- host vault persistence
- network-type based startup selection

In other words:

- the Fabric adapter exists
- the host bootstrap policy around it is still not fully normalized

## Target Network Model

The target model is not “Fabric or Pontus-X by accident”.

It is:

- `fabric-only`
- `pontusx-only`
- `dual`

selected explicitly by hosting-operator network support.

## Role Of Fabric In The Target Model

Fabric is the direct network/ledger integration path.

Expected properties:

- direct adapter inside GW CORE
- direct network bootstrap owned by the hosting operator
- direct cryptographic/runtime material handling
- direct ledger writes for hosted tenants

The hosting-operator bootstrap path for Fabric is the direct path described in
the audit docs:

- resolve host material from env or vault
- otherwise bootstrap via `_exchange -> _dcr -> _csr`
- persist resulting host artifacts
- then operate through Fabric

## Role Of Pontus-X In The Target Model

Pontus-X is currently not documented as the same direct certificate-based path
as Fabric.

At the current phase, the expected model is:

- mediated through the `globaldatacare.es` backend
- authenticated via API key or equivalent technical credential
- used to publish on behalf of:
  - the hosting service
  - or a hosted tenant

This is especially relevant for European `DigitalTwin` or dataset publication
flows.

So the practical distinction is:

- Fabric: direct adapter and direct technical network operation
- Pontus-X: mediated backend adapter and delegated publication path

## Important Modeling Rule

Do not collapse the two models into one fake abstraction.

Use one abstraction layer, but preserve the operational differences:

- Fabric uses direct ledger connectivity and its own runtime cryptography
- Pontus-X currently uses intermediary backend semantics and API-key style
  authentication

The adapter interface may be shared, but the bootstrap contract is not
identical.

## Concrete Reuse Rule

For Fabric:

- reuse `src/blockchain/fabric/v3`
- reuse `CredentialLedgerAdapterFabric`
- do not create a second parallel Fabric adapter unless the existing one is
  being intentionally replaced

For Pontus-X:

- add a new explicit adapter
- do not fake Pontus-X support by overloading Fabric assumptions

## Implementation Tasks

### 1. Fabric inventory cleanup

- document exactly which `fabric/v3` modules are canonical
- identify which files are legacy but still authoritative
- define which entrypoints GW CORE should officially use

### 2. Host bootstrap integration for Fabric

- connect hosting-operator startup to the existing Fabric adapter stack
- make the startup order explicit:
  - env
  - vault
  - ICA bootstrap
- persist resulting host certificate/runtime artifacts in vault

### 3. Network-type configuration

Introduce canonical host-level configuration that distinguishes:

- business sector
- network type

Expected values:

- `fabric`
- `pontusx`
- `dual`

### 4. Pontus-X adapter

Add a new mediated adapter for Pontus-X once the backend API contract is
available.

The adapter must support publication context for:

- hosting service
- hosted tenant

### 5. Dual-mode runtime policy

Define how GW behaves when both adapters are enabled:

- which operations go to Fabric
- which operations go to Pontus-X
- which operations may go to both
- what happens when one adapter is degraded

## Audit Statements

1. Fabric support already exists in GW CORE and must be reused.
2. Fabric remains part of the architectural baseline because of the SEDIA
   memory and planning assumptions.
3. Pontus-X support must be additive and adapter-specific.
4. European `DigitalTwin` publication may use Pontus-X while Fabric remains
   supported for other trust/ledger responsibilities.
5. The correct next step is integration and normalization, not reinvention of a
   Fabric adapter from scratch.

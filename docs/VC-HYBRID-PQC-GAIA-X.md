## Hybrid PQC + Gaia-X VC (VC v2)

This document describes the hybrid signing model used for Gaia-X IAM 25.11 VCs in this repo. The goal is to remain compatible with Gaia-X legacy ECDSA requirements while adding an optional post-quantum proof.

References:
- Gaia-X IAM 25.11 Credential Format: https://docs.gaia-x.eu/technical-committee/identity-credential-access-management/25.11/credential_format/
- Gaia-X IAM 25.11 Status List: https://docs.gaia-x.eu/technical-committee/identity-credential-access-management/25.11/standards/Status%20List
- RFC7518 (JWA): https://www.rfc-editor.org/rfc/rfc7518
- RFC7517 (JWK): https://www.rfc-editor.org/rfc/rfc7517

### Model: self-signed + ICA attested
We keep a clear split between the participant's self-signed credential and the ICA attestation, but both can be represented as a single VC with multiple proofs:
- Participant self-signed proof(s):
  - Legacy ECDSA (ES256 or ES384) proof for Gaia-X compatibility.
  - Optional ML-DSA proof for post-quantum assurances.
- ICA attestation proof(s) over the same VC:
  - Legacy ECDSA proof (ES256 or ES384) embedding x5c/x5u per Gaia-X.
  - Optional ML-DSA proof by the ICA.

Recommended storage and API shape:
- Persist the raw self-signed VC as the "self-description" (participant only).
- Persist the ICA-attested VC as the "legal participant" VC (same subject, same data, extra proof(s)).
- Both are VC v2 and include StatusList2021.

This avoids mismatched subject data between the participant and the ICA. It also keeps a clear audit trail (self-signed first, then ICA attestation).

### Claim mapping (internal -> Gaia-X)
Internal `meta.claims` use reversed-DNS keys (e.g., `org.schema.Organization.legalName`). Gaia-X VCs MUST expose the Gaia-X credentialSubject profile (e.g., `gx:LegalParticipant`, `gx:legalName`, `gx:legalRegistrationNumber`). For this reason, VCs are generated by mapping internal claims into the Gaia-X subject structure; raw `claims` are not inserted directly into `credentialSubject`.

### Proof requirements for Gaia-X legacy compatibility
Gaia-X 25.11 expects legacy ECDSA-compatible proofs aligned with JWA (RFC7518). The key material used for those proofs must expose its X.509 chain:
- The public JWK MUST include either `x5c` or `x5u` per RFC7517.
- `x5u` must resolve to an X.509 certificate chain (`.crt`, `.pem`, `.der`, or `.p7b`) for a Gaia-X Trust Anchor eligible for the claims.

In this repo:
- Legacy signing uses `LEGACY_SIGN_ALG` (ES256 or ES384). If missing or invalid, default is ES384.
- The `/.well-known/x509.der` endpoint returns a concatenated DER chain (leaf + ICA + root).
- The JWK for legacy ECDSA includes `x5u` and `x5c` when DER is available.

### VC v2 + StatusList2021
All Gaia-X VCs are emitted with VC v2 semantics:
- `@context` includes the VC v2 context.
- `validFrom` is used instead of `issuanceDate`.
- `credentialStatus` is a StatusList2021 entry.
- A StatusList2021 VC is generated on demand and signed similarly.

### Endpoints
- `/.well-known/jwks.json`: includes `x5u`/`x5c` for the legacy key when DER is available.
- `/.well-known/x509.der`: returns concatenated DER chain (leaf + ICA + root).
- `/.well-known/status-list.json`: StatusList2021 VC signed on demand.
- `/.well-known/legal-participant.vc.json`: Gaia-X Legal Participant VC with `credentialStatus` and multiple proofs.
- `/.well-known/vc.json`: Legacy alias for `legal-participant.vc.json`.

The well-known endpoints return stored artifacts from the entity configuration; they do not generate credentials on demand.

### SMART token legacy signing
For SMART on FHIR compatibility, tokens default to legacy ECDSA signing when a legacy key exists. Set `SMART_TOKEN_LEGACY=false` to force ML-DSA signing.
- `/{role}/.well-known/*`: same endpoints for `role=ca|ica` when `LOCAL_SERVICE_ROLE` enables them.
- `/{role}/identity/gaia-x/credential/_sign`: async signer endpoint for CA/ICA.

### Environment variables (host or defaults)
- `LEGACY_SIGN_ALG=ES384` (default if missing or invalid)
- `LEGACY_X509_DER_BASE64` (leaf DER base64)
- `LEGACY_X509_CHAIN_BASE64` (ICA,root DER base64, comma-separated)
- `LOCAL_SERVICE_ROLE=HOST,ICA,CA`
- `LOCAL_CA_ARTIFACTS_DIR` (optional override)
- `LOCAL_ICA_ARTIFACTS_DIR` (optional override)

### PKI generation for test-network
The generator produces Root CA, ICA, and Host MSP artifacts (includes `x509-chain.der`). It uses P-384 when `LEGACY_SIGN_ALG=ES384`:
```
npm run pki:generate
```
With `.env.local`:
```
DOTENV_CONFIG_PATH=.env.local npm run pki:generate
```

Required variables (see `env.example`):
```
DOMAIN_ROOT_CA, DOMAIN_ICA, ORG_JURISDICTION, ORG_CITY, ORG_STREET, ORG_POSTAL_CODE,
ROOT_CA_LEGAL_NUMBER, ROOT_CA_ORG_NAME, ROOT_CA_SEED,
ICA_LEGAL_NUMBER, ICA_ORG_NAME, ICA_SEED,
HOST_LEGAL_NUMBER, HOST_ORG_NAME, HOST_DOMAIN, HOST_CN, HOST_SEED
```

### Notes
- `x509.der` is concatenated to let verifiers evaluate the full chain.
- Legacy ES and ML-DSA proofs can coexist in `proof[]` for the participant and the ICA.
- The ICA proof includes `x5c` in the JWS header (and optionally `x5u`), so the chain is available in-band and by URL.

# TODO: Tenant Identifier And Vault Migration (GW CORE v2.0)

Status: Planned for `v2.0`

## Why this is deferred

`GW CORE v1.x` still contains compatibility behavior that couples hosted-tenant runtime resolution to an internal `alternateName` alias in several paths. Replacing that model immediately would break existing vault ids, routes, tests, and onboarding flows.

## Current v1.x contract

- For legal organizations, the canonical external identifier is `Organization.identifier.value`.
- In current ICA-aligned examples, that value is the legal/tax identifier returned by the ICA flow.
- If a legal-organization registration omits `Organization.alternateName`, GW CORE derives an internal compatibility alias from `Organization.identifier.value`.
- `Organization.alternateName` is reserved in public examples for individual/family-style onboarding.

## v2.0 target

1. Build logical tenant/vault identity from canonical `Organization.identifier.value`, not from `alternateName`.
2. Remove legal-organization dependence on `alternateName` in:
   - onboarding
   - vault id construction
   - cache lookup
   - worker/runtime routing
   - host registry persistence
3. Keep `alternateName` only as an optional display or nickname-style claim where that concept is still valid.
4. Define jurisdiction-aware identifier policy so `identifier.value` is not hardcoded to `taxId` in countries where the legal registration id differs from tax ids.

## Jurisdiction policy to formalize in v2.0

- `ES`: `taxId` may map directly to `Organization.identifier.value`.
- `CA-BC`: provincial incorporation/registration number should be the legal primary identifier; federal BN/GST/import-export ids remain secondary tax/program identifiers.
- Additional jurisdictions must document:
  - legal primary identifier
  - tax identifier(s)
  - whether they are equal or distinct

## Required implementation audit before migration

- `src/utils/tenant.ts`
- `src/managers/HostingManager.ts`
- `src/managers/TenantsCacheManager.ts`
- `src/routes/api.ts`
- `src/worker.ts`
- all managers/routes that still call `getTenantVaultId(sector, tenantId)` assuming `tenantId == alternateName`

## Non-goals for this TODO

- Syncing these changes into `GW UNID`
- Changing current `v1.x` stored vault ids in-place without a migration plan
- Rewriting all historical docs/examples that intentionally describe current compatibility behavior

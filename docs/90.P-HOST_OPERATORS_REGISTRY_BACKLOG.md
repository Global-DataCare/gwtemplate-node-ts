# Node Operator Catalog Backlog

Status: Transitional

## Goal

Implement host-side node operator discovery and hosted-tenant publication in
`gwtemplate-node-ts` without conflating it with tenant sector catalog
publication.

## Work Packages

### 1. Projection model

- Add `HostedTenantCatalogRecord` type.
- Add storage adapter or cache methods for hosted tenant catalog projection
  records if materialization is needed.
- Add cache loader / invalidation strategy similar to tenant discovery cache.

Likely areas:

- `src/services/*`
- tenant cache / registry helpers
- storage adapter layer used by discovery/runtime config

### 2. Activation-driven publication rule

- Derive host catalog publication from activated tenant organizations.
- Use `Service.serviceType` as the source of truth for discoverable service
  capabilities.
- Exclude reader-only capabilities from service autodiscovery publication.
- Decide explicitly whether `DigitalTwinIndex` is externally discoverable or
  internal-only.

### 3. Host catalog generation

- Introduce canonical host endpoint:
  - `GET /.well-known/dcat3/catalog`
- Keep compatibility alias:
  - `GET /dcat3/catalog/dcat.json`
- Refactor catalog builder to read from activated tenant service capability
  projection instead of directly projecting only raw onboarding claims.

Expected output:

- host-level `dcat:Catalog`
- entries for the node operator itself and/or hosted tenant organizations,
  represented through `dcat:Dataset`, `dcat:DataService`, or a hybrid shape as
  decided by implementation
- only tenants that actually offer services must be included; reader-only
  tenants must be excluded

### 4. Host DID publication

- Publish host `CatalogService` DID entry for `/.well-known/dcat3/catalog`.
- Document that it is a node-operator discovery service.
- Keep DID service publication as the primary discovery surface.
- Treat `/.well-known/api-catalog` only as a future complementary TODO.

Likely areas:

- DID service config utils
- discovery service docs
- DID compliance tests

### 5. Tenant sector catalog canonicalization

- Keep the current tenant DSP request binding as the implemented contract:
  - `POST /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/request`
  - `GET /{tenantId}/cds-{jurisdiction}/v1/{sector}/dcat3/catalog/datasets/{id}`
- If a read-only tenant `/catalog` resource is added later, document it as an
  additional artifact rather than replacing the current DID `CatalogService`
  binding.
- Explicitly separate host catalog implementation from tenant sector catalog
  implementation.

### 6. Optional HealthDCAT-AP Release 7 tenant profile

- Keep `DCAT3` as the operational baseline serializer/profile.
- Apply `HealthDCAT-AP Release 7` only when a deployment explicitly chooses an
  additional health-oriented sector profile.
- Move shared types/builders into `gdc-common-utils-ts` once stable.

### 7. Documentation

- Update:
  - `docs/04-DEEP-DIVES/04.D-DISCOVERY-SERVICES.md`
  - `docs/02-API-AND-ENDPOINTS/02.E-DATASPACE-DID-SERVICES.md`
  - `docs/02-API-AND-ENDPOINTS/02.A-API-ENDPOINTS.md`
- Link this backlog from `docs/README.md`.

### 8. Tests

- Add integration test for `/.well-known/dcat3/catalog`.
- Keep `/dcat3/catalog/dcat.json` as alias test.
- Add tests that only provider-style service capabilities are published.
- Add tests that reader-only tenants are excluded.
- Add host DID service compliance tests for host catalog endpoint.
- Add tenant sector catalog tests for the current tenant DSP request binding.
- Add profile tests only for deployments that opt into `HealthDCAT-AP Release 7`.

Likely existing tests to extend:

- `src/__tests__/integration/wellKnownApi.test.ts`
- `src/__tests__/unit/utils/dataspace.did-services.compliance.test.ts`
- `src/__tests__/unit/utils/swagger-spec.test.ts`

## Suggested Implementation Order

1. Hosted-tenant projection type + storage/cache if needed
2. Activation-driven publication rule
3. Host `/.well-known/dcat3/catalog`
4. Host DID `CatalogService`
5. Decide whether a tenant read-only `/catalog` artifact is needed in addition
   to the current DSP request binding
6. Optional `HealthDCAT-AP Release 7` tenant serializer/profile
7. `/.well-known/api-catalog` as optional complementary TODO
8. ICA sync/attestation as later phase

## Non-Goals

- Do not move the whole problem into ICA first.
- Do not treat node operator discovery catalog as equivalent to tenant sector
  HealthDCAT catalogs.
- Do not force tenant multi-sector catalogs into the host root `/.well-known`
  namespace.
- Do not publish reader-only tenants in service autodiscovery.
- Do not describe project-local `/dcat3/...` routes as if they were the DSP
  standard path bindings.
